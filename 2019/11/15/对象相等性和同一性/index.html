引入概念

很多时候我们写程序会比较两个变量，用作搜索、排序或者其他。
比较的基准可以分为两种：同一和相等。
下面解释下：

  • 同一性：两个变量都是引用同一个对象。
  • 相等性：两个变量的各个成员和字段的值相等。

同一自然相等，但是相等不一定同一

C#提供的方法

在编写程序时，常常需要进行相等性或者同一性的检查，C#的System.Object类型提供了Equals虚方法：

public class Object
{
    public virtual Boolean Equals(Object obj)
    {
        if(this == obj) return true;
        return false;
    }
}

注意，这里仅是比较两个变量的引用相等，也就是是否是同一个对象，所以是同一性的判断。
显然默认的Equals方法是不够满足使用的，我们常需要重写Equals方法来判断相等性（这章只介绍相等性和同一性，关于具体的重写策略，篇幅较长，会在后面的文章补充）。也正是由于Equals方法能被重写，因此一般不再用它测试同一性，Object提供了专门的静态方法ReferenceEquals：

public class Object
{
    public static Boolean ReferenceEquals(Object objA, Object objB)
    {
        return (objA == objB);
    }
}

检查同一性务必调用ReferenceEquals，不应该使用==操作符或者Equals，后两者可能被重载或者重写。

对于值类型，它们的基类System.ValueType就重写了Equals方法，并且正确地实现了相等性的检查，实现如下：

  1. 如果obj实参为null，返回false
  2. 如果this和obj实参引用不同类型的对象，返回false。
  3. 针对类型定义的每个实例字段，都将this对象中的值和obj对象中的值进行比较（通过调用字段的Equals方法）。任何字段不相等，就返回false。
  4. 返回true。不用调用ValueType的基类Object的Equals方法，因为Object没有定义任何字段。

对于第3步骤，ValueType的Equals方法是利用反射实现的。由于CLR反射机制慢，我们定义自己的值类型时，应该重写Equals方法来比较值相等性，从而提高性能。注意不要调用base.Equals，我们的目的就是避免调用反射实现的Equals。

重写Equals方法的注意点

定义自己的类型时，重写Equals要符合相等性的4个特征：

  • 必须自反；x.Equals(x)肯定为true。
  • 必须对称；x.Equals(y)和y.Equals(x)结果一致。
  • 必须可传递；x.Equals(y)和y.Equals(z)都为true，则x.Equals(z)必定为true（注意都为false则不成立）。
  • 必须一致；比较的两个值不变，结果不能改变。

重写的规范：

  • 让类型实现System.IEquatable接口的Equals方法，类型安全的Equals方法在内部调用。
  • 实现参数为Object类型的Equals方法，内部调用类型安全的Equals方法。
  • 如果需要，重载==、!=等操作符，在内部调用类型安全的Equals方法。

如果类型需要为排序而比较，类型还应该实现System.IComparable的CompareTo方法和System.IComparable的类型安全的CompareTo方法。还可考虑重载比较操作符（<、>=等），在内部调用类型安全的CompareTo方法。
