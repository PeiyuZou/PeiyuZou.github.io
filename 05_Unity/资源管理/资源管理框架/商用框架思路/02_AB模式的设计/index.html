    02.AB模式的设计

    一、设计总览¶

    Ab 模式的核心是 AbManager 类，他是 IManageResource 的一种策略实现，它的主要职责：

    • 持有主清单（CustomMainManifest manifest）
    • 把"逻辑路径"翻译成真实文件路径（包内 / 热更 / AAB / 平台差异）
    • 维护 AB 包索引表 mAbDict 和 回收站 mAbRecycleBin
    • 调度 异步加载（限制并发数、Update 轮询完成状态）
    • 不亲自执行加载流程——具体加载放在 AbInfo 里

    总的来说，每一个 AB 包对应一个 AbInfo，真正的加载是 AbInfo 完成的，AbManager 负责调度和管理所有 AB 包

    二、初始化的重点任务：把 Manifest 清单文件加载起来¶

    AB 模式的 "第一加载" 永远是 Manifest，只有读到 Manifest，我们才知道：

    • 项目里有哪些 AB 包
    • 每个 AB 包里有哪些 Asset
    • 每个 AB 依赖哪些其它 AB
    初始化加载清单文件的伪代码
    if (hotfix && File.Exists(hotfixPath)) // 有热更的情况
{
    originalManifestBundle?.Unload(true);
    AssetBundle manifestBundle = AssetBundle.LoadFromFile(hotfixPath, 0, ASSETBUNDLE_OFFSET);
    manifest = manifestBundle.LoadAllAssets<CustomMainManifest>()[0];
    manifestBundle.Unload(false);
}
else                                   // 包内回退
{
    CustomMainManifest originalManifest =
        originalManifestBundle.LoadAllAssets<CustomMainManifest>()[0];
    manifest = originalManifest;
    originalManifestBundle.Unload(false);
}

    在需要加载一个AB包的时候，通过路径优先级来获取文件路径，这样热更的时候会优先读热更路径下的文件

    获取AB包路径的优先级
    public string GetFilePath(CustomAssetBundleManifest abData)
{
    string finalPath;
    if (!string.IsNullOrEmpty(abData.HotfixName))      // 1. 热更
    {
        finalPath = ToHotfixFinalPath(abData.HotfixName);
        //因 包内->热更1->热更2(热更2==包内)。SDK下载热更时不会下载文件`热更2`。同时存在的APK包也有多个
        if (File.Exists(finalPath)) return finalPath;
    }
    if (Application.platform == RuntimePlatform.Android && GetFileAABPath != null)
    {
        finalPath = GetFileAABPath(abData.Name);        // 2. AAB
        if (!string.IsNullOrEmpty(finalPath)) return finalPath;
    }
    finalPath = ToOriginalFinalPath(abData.Name);       // 3. 包内
    return finalPath;
}

    注意踩坑

    特别留意源码里这条注释："包内→热更1→热更2(热更2==包内)。SDK下载热更时不会下载文件 热更2"——这是真实踩过的坑： 某些资源被回滚成跟包内一样的内容，差量下载就不会再下，于是 File.Exists(hotfixPath) 是 false，必须 fallback 到包内。

    三、为什么写了一个 CustomMainManifest ？¶

    Unity 内置的 AssetBundleManifest 提供了：

    • 所有 AB 包名列表（带 hash）
    • 每个 AB 的依赖列表

    但它不提供：

    • "Asset 路径 → AB 包"反查（必须遍历）
    • "逻辑名 → 加载名"映射
    • 热更版本号、自定义元数据

    除了 CustomMainManifest 还有一个 CustomAssetBundleManifest，它使用 int 索引，记录单个 AB 包引用的 Asset 的索引和它依赖的 AB 包的索引，这样的作用是运行时访问 Asset 和依赖包都更快，并且序列化的数据更小。但是缺点是需要在构建期保证索引稳定，不能随意排序。

    异步加载请求不直接执行，而是加入链表
    internal void LoadAbAsync(AbInfo abInfo)
{
    mCurAbLoadingCount++;
    currentLoadingAbInfoList.AddLast(abInfo.LoadAssetBundleAsync());
}
    收割：每帧 Update 看谁完成了
    protected void Update()
{
    LinkedListNode<AbInfo> abNode = currentLoadingAbInfoList.First;
    while (abNode != null)
    {
        var next = abNode.Next;
        if (abNode.Value.LoadAssetBundleFinish())
            currentLoadingAbInfoList.Remove(abNode);
        abNode = next;
    }

    LinkedListNode<AssetInfo> assetNode = currentLoadingAssetInfoList.First;
    while (assetNode != null)
    {
        var next = assetNode.Next;
        if (assetNode.Value.LoadAssetSyncNewFinish())
        {
            AbInfo abInfo = mAbDict[assetNode.Value.BundleName];
            abInfo.CompleteLoadAsset(assetNode.Value);
            currentLoadingAssetInfoList.Remove(assetNode);
        }
        assetNode = next;
    }

}

    五、回收站机制：延迟卸载的智慧¶

    当一个 AB 的引用计数变成 0 时，不立即卸载，而是加入回收站
    public void AddToRecycleBin(string s)
{
    if (!mAbRecycleBin.ContainsKey(s))
        mAbRecycleBin.Add(s, mCheckElapsed + recycleLife);  // recycleLife = 2.333f
}
    延迟卸载
    // Update() 里每秒检查一次
mCheckElapsed += Time.unscaledDeltaTime;
if (mCheckElapsed - mLastCheckTime >= checkLifeInterval) // checkLifeInterval = 1f
{
    mLastCheckTime = mCheckElapsed;
    List<string> toRemove = new List<string>();
    foreach (var lifeInfo in mAbRecycleBin)
        if (lifeInfo.Value <= mCheckElapsed)
            toRemove.Add(lifeInfo.Key);
    foreach (string s in toRemove)
    {
        mAbRecycleBin.Remove(s);
        mAbDict[s].TryRelease(true);
    }
}

    为什么这样设计？考虑一个场景：

    引用计数归零，但又马上加载
    第 N 帧：上一个 UI 关闭 → AB 引用变 0 → 想卸
第 N+1 帧：玩家立刻又点开同一个 UI → 又要加载这个 AB

    如果引用归零立刻卸载，会出现 "刚卸又加载" 的颠簸：磁盘 IO 两遍、贴图重新上传 GPU 两遍。 引入 recycleLife 的延迟，给 "近期可能复用" 留一个窗口。

    为什么是 2.333f 这个怪数？仅仅是一个根据经验得到的值。既要覆盖关 UI 又秒打开的场景，又不至于让显存堆太久。

    从回收站复活
    internal AbInfo GetAbInfo(...)
{
    ...
    if (mAbRecycleBin.ContainsKey(abPath))
    {
        // 在回收列表中，说明之前刚刚归零，现在又被需要了
        mAbRecycleBin.Remove(abPath);
    }
    return abInfo;
}

    这是延迟卸载机制的关键收益点：被 "复活" 的 AB 直接取回，无需重新加载。
