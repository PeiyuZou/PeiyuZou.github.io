1. 这一层是干什么的

体素大世界最终要落到一个具体的场景类上，这个类就是 MCScene。但 MCScene 不是凭空开始的，它继承自一个更底层的基类 UniversalScene。本篇只讲这个基类。

先看它的定义（Assets/Scripts/Engine/UniversalScene/UniversalScene.cs:21）：

public class UniversalScene : MonoBehaviour, ILoadingStateSwitch, IRefreshable

它是一个 MonoBehaviour（所以它挂在 GameObject 上，能收到 Unity 的 Awake/Start/Update/OnEnable/OnDisable 生命周期），同时实现了两个接口：

  • ILoadingStateSwitch：能切换"加载中/加载完成"两种状态（定义在 Assets/Scripts/Engine/UniversalScene/Components/SceneObjects/Entity/Interface/ILoadingState.cs:105，只有一个方法 SwitchToLoadingState(bool)）。加载中时调度器会改变每帧加载上限，让资源抢着进来；加载完成后恢复常态。
  • IRefreshable：能被通知"刷新一次"（断线重连后用），定义在 Assets/Scripts/Engine/UniversalScene/Features/IRefreshable.cs:6。

它解决了什么问题

体素大世界和普通场景不一样：它没有"一整块烘焙好的地图"，而是围绕一个中心点动态加载/卸载一堆 chunk 和物件。这就要求场景类本身具备几项通用能力：

  • 可加载、可卸载：场景有自己的配置文件、资源、生命周期，能创建出来也能干净地拆掉。
  • 自带一个工作线程：chunk 的网格构建、数据解压等耗时计算不能卡 Unity 主线程，要丢到后台线程跑。
  • 用组件组织能力：物件管理、MC 管理、遮挡剔除……这些都是独立模块，应该像挂组件一样挂到场景上，而不是全堆在场景类里。
  • 有 LOD 更新入口：每帧根据相机位置更新物件的 LOD 级别。
  • 有加载状态机：能记录"chunk 数据好了没、物件好了没、地形 patch 好了没"，全好了才通知上层"场景加载完成"。
  • 有一个"扫描中心"概念：接收一个坐标，判断要不要触发新一轮重加载。

UniversalScene 把这些能力统一下沉到基类，让上层的 MCScene 只关心"体素"那一摊事（chunk、block、MCWorld），不用再操心线程、组件、加载状态这些基础设施。这就是"场景基类"这个抽象的价值。

一句话定位：UniversalScene = 一个自带工作线程、用组件拼装、按中心点驱动重加载的 MonoBehaviour 场景骨架。MCScene 在它之上长出体素能力。

这里有几个设计细节值得点一下：

  • 用 double 存坐标：m_Location 是 YKVector3d（double3）。体素大世界尺度极大，单精度 float 在远离原点处会出现精度抖动，所以扫描中心用 double。初始值是 Double.MinValue 三件套，用来表示"还没设过"。
  • 三个 Manager 字段是子类赋值的：UniversalScene 自己只声明了 m_SceneObjectManager（在 Initialize 里注册），m_MCManager / m_MCOCManager 留给 MCScene 在它自己的 Initialize 里注册。基类只提供"持有 + 暴露"的载体。
  • new 关键字修饰 GetComponent<T>（行 162）：因为 MonoBehaviour 本身就有个 GetComponent，这里用 new 隐藏掉父类那个，改成查自己的组件字典。所以场景上调用 scene.GetComponent<SceneObjectManager>() 拿的是 SceneComponent 体系里的组件，不是 Unity 的 Component。

3. SceneComponent 组件化机制

3.1 SceneComponent 是什么

SceneComponent 是一个抽象普通类，不是 MonoBehaviour（Assets/Scripts/Engine/UniversalScene/SceneComponent.cs:3）。它不挂在 GameObject 上，而是被场景持有：

public abstract class SceneComponent
{
    public UniversalScene Scene { get; }

    protected SceneComponent(UniversalScene scene)
    {
        Scene = scene;
    }

    public virtual void Start() { }
    public virtual void Update() { }
    public virtual void OnEnable() { }
    public virtual void OnDisable() { }
    public virtual void OnDestroy() { }
    public virtual void OnDrawGizmos() { }
}

它通过构造函数把"我属于哪个场景"拿进来（Scene 属性只读），并且仿照 Unity 的生命周期提供了一组虚方法。注意这些方法不是 Unity 自动调的（因为它不是 MonoBehaviour），而是 UniversalScene 在自己的 Unity 生命周期里手动转发给每个组件的。

3.2 注册与获取

组件存在 m_Components 这个 Dictionary<Type, SceneComponent> 里，key 是组件类型。注册和获取两个方法很轻（行 154-170）：

public T RegisterComponent<T>() where T : SceneComponent
{
    SceneComponent component = SceneUtils.CreateInstance<T>(this);  // 反射创建，把 this 传进构造
    m_Components.Add(typeof(T), component);
    return (T)component;
}

public new T GetComponent<T>() where T : SceneComponent
{
    if (m_Components.TryGetValue(typeof(T), out var component))
        return (T)component;
    return null;
}

SceneUtils.CreateInstance<T>(this) 本质是 Activator.CreateInstance(typeof(T), this)（Assets/Scripts/Engine/UniversalScene/Utils/SceneUtils.cs:13），即用反射调用 SceneComponent 子类的构造，把当前场景传进去。所以每个 SceneComponent 子类的构造必须接收一个 UniversalScene 参数，例如 SceneObjectManager（Assets/Scripts/Engine/UniversalScene/Components/SceneObjects/SceneObjectManager.cs:67）：

public SceneObjectManager(UniversalScene scene) : base(scene)
{
    SceneEntity.OnUpdateTransform -= UpdateTransformAndQuadTree;
    SceneEntity.OnUpdateTransform += UpdateTransformAndQuadTree;
}
注意：同一个类型只能注册一次（用的是 Dictionary.Add，重复注册会抛异常）。组件以类型为 key，所以"场景上只有一个 SceneObjectManager"是靠字典结构保证的。

3.3 生命周期怎么和 Scene 绑定

UniversalScene 在自己的 Unity 生命周期里逐个转发调用给组件。几处关键转发：

// Assets/Scripts/Engine/UniversalScene/UniversalScene.cs
protected void Start()                          // 行 207
{
    foreach (var component in m_Components.Values)
        component.Start();
}

public virtual void Update()                    // 行 304
{
    RefreshThreadTask(m_UnityExecuteSyncBuffer); // 先跑主线程任务
    ...
    foreach (var component in m_Components.Values)
        component.Update();                      // 再跑每个组件的 Update
    Scheduler.Update();
    ...
}

protected virtual void OnEnable()  / OnDisable() // 行 288 / 296 —— 同样遍历转发

卸载时则逆序销毁（行 344 UnLoad）：

var components = m_Components.Values.ToList();
for (int i = components.Count - 1; i >= 0; i--)
{
    try { components[i].OnDestroy(); }
    catch (Exception e) { Debug.LogError("场景卸载失败:" + components[i].GetType() + "\n" + e); }
}
m_Components.Clear();

逆序是为了让后注册的组件（更上层的）先销毁，避免底层组件先被拆掉导致上层引用空。每个组件的 OnDestroy 包了 try-catch，单个组件销毁失败不会阻塞其它组件和场景卸载。

3.4 谁是 SceneComponent

看 MCScene.Initialize（Assets/Scripts/Core/MCCore/Scene/MCScene.cs:34）注册了哪些组件，就知道这一层是怎么"拼"出来的：

m_MCManager = RegisterComponent<MCManager>();        // 或 MCNetworkManager / MCModifyManager
m_MCBlockDecalManager = RegisterComponent<MCBlockDecalManager>();
RegisterComponent<ClientScannerManager>();
RegisterComponent<SalePartnerScannerManager>();
m_StaticSceneOcclusionCullingManager = RegisterComponent<StaticSceneOcclusionCullingManager>();
m_MCOCManager = RegisterComponent<MCOCManager>();
base.Initialize(args);   // 基类里还会注册 SceneObjectManager

而 UniversalScene.Initialize 自己注册的是物件总管（行 186）：

m_SceneObjectManager = RegisterComponent<SceneObjectManager>();

所以一个 MCScene 实例上挂着这些组件：SceneObjectManager、MCManager（或其网络/修改变体）、MCBlockDecalManager、ClientScannerManager、SalePartnerScannerManager、StaticSceneOcclusionCullingManager、MCOCManager。每个组件各管一摊，场景类只负责把它们串起来、转发生命周期。

小结：SceneComponent 是一套自建的轻量组件体系，不走 Unity 的 MonoBehaviour。好处是创建/销毁/转发都自己控制，方便在工作线程里统一驱动（见下一节），也不需要每个组件都挂 GameObject。

4. 自有工作线程与线程任务分发（重点）

这是 UniversalScene 最关键的设计之一，要讲透。

4.1 为什么体素大世界要单独开线程

体素大世界每时每刻都在做大量计算：玩家一移动，围绕新中心点要扫描该加载哪些 chunk、要解压 chunk 数据、要构建 chunk 的网格、要把物件塞进四叉树……这些操作数据量大、计算密集。如果全在 Unity 主线程做，主线程每帧只有十几毫秒，根本扛不住，画面会卡死。

所以 UniversalScene 自己起了一个后台工作线程（不是 Unity 主线程），把那些"不需要碰 Unity API 的纯数据/纯计算"活儿丢过去跑，主线程只负责"需要 Unity API 的那部分"（创建 GameObject、挂材质、改 Transform 等）和最终渲染。

4.2 线程的启动与停止

工作线程在场景资源加载完成时启动。调用链是 OnLoad → Load → ThreadStart（行 680-688）：

// UniversalScene.cs:680
protected virtual void Load()
{
    if (m_EnableThreadMode)
        ThreadStart();
}

// UniversalScene.cs:217
private void ThreadStart()
{
    if (!m_IsThreadRunning)
    {
        m_IsThreadExit = false;
        m_IsThreadRunning = true;
        Thread t = new Thread(ThreadRunnable);
        t.Name = "Universal Scene Worker Thread#" + t.ManagedThreadId;
        t.Start();
    }
}

停止在 UnLoad 里调用 ThreadEnd（行 229）：

protected virtual void ThreadEnd()
{
    if (m_IsThreadRunning)
    {
        m_IsThreadRunning = false;          // 先把循环条件翻成 false
        while (!m_IsThreadExit)              // 等工作线程自己跑出循环、把 m_IsThreadExit 置 true
            Thread.Yield();
    }
}

这是个标准的"标志位 + 自旋等待"停线程方式：m_IsThreadRunning 和 m_IsThreadExit 都是 volatile，保证主线程写、子线程读的可见性。主线程设 m_IsThreadRunning = false 后就 Thread.Yield() 等子线程自己也跑完最后一轮、把 m_IsThreadExit 置 true，再继续往下销毁，避免销毁时子线程还在跑导致访问已释放的数据。

4.3 工作线程主循环

// UniversalScene.cs:241
protected virtual void ThreadRunnable()
{
    while (m_IsThreadRunning)
    {
        try { ThreadUpdate(); }
        catch (Exception e)
        {
            ExecuteInUnity(new ThreadTask(e, exception => { Debug.LogError(e); }));
        }
        Thread.Sleep(ThreadTriggerInterval);   // 默认 100ms 跑一轮
    }
    m_IsThreadExit = true;
}

循环体很简单：每轮跑一次 ThreadUpdate()，然后睡 100ms（THREAD_TRIGGER_INTERVAL_TIME = 100，行 56）。子线程里如果抛异常，不直接 Debug.LogError（因为 Debug.Log 在子线程虽然能调但有些 Unity API 不行），而是把异常包成一个任务丢回主线程打印。

MCScene 重写了这个方法（Assets/Scripts/Core/MCCore/Scene/MCScene.cs:413），区别是：场景还在加载中时用 Thread.Yield() 代替 Thread.Sleep，让工作线程尽量满负荷跑，加快首帧加载。

4.4 ThreadUpdate 做什么

// UniversalScene.cs:270
protected void ThreadUpdate()
{
    RefreshThreadTask(m_ThreadExecuteSyncBuffer);   // 1. 执行后台任务队列里的任务

    foreach (var component in m_Components.Values)
    {
        if (component is ISchedulable schedulable)  // 2. 让调度器驱动"可调度"组件扫描 patch
            Scheduler.Execute(schedulable);

        if (component is IThreadRunnable threadRunnable)  // 3. 让"可线程运行"组件跑自己的后台逻辑
            threadRunnable.ThreadUpdate();
    }
}

三件事：

  1. 消费后台任务队列：把别人通过 ExecuteInThread 丢进来的任务跑掉。
  2. 调度器驱动：实现了 ISchedulable 的组件（比如 MCManager）交给 TiledScheduler.Execute，让它根据当前 m_Location 扫描该加载/卸载哪些 patch（Assets/Scripts/Engine/UniversalScene/Scheduler/TiledScheduler.cs:49，注释里写明 Executed in scene worker thread）。
  3. 组件后台 tick：实现了 IThreadRunnable 的组件（比如 SceneObjectManager）调它的 ThreadUpdate() 跑后台逻辑。IThreadRunnable 接口只有一个 ThreadUpdate() 方法（Assets/Scripts/Engine/UniversalScene/Features/IThreadRunnable.cs:3）。

4.5 ExecuteInThread / ExecuteInUnity —— 两个队列与双缓冲

这是这对 API 的核心。它们都是"入队"操作，本身不阻塞、不切线程，只是把一个任务塞进队列，等对应线程下一轮来取（行 434-442）：

public void ExecuteInThread(ThreadTask task)   // 丢给后台线程跑
{
    m_ThreadExecuteSyncBuffer.Enqueue(task);
}

public void ExecuteInUnity(ThreadTask task)    // 丢回主线程跑
{
    m_UnityExecuteSyncBuffer.Enqueue(task);
}

两个队列都是 YoukiaEngineCore.ConcurrentQueue<ThreadTask>（线程安全的并发队列）。消费端用了一个"双缓冲切换"的小技巧（行 261）：

private void RefreshThreadTask(YoukiaEngineCore.ConcurrentQueue<ThreadTask> tasks)
{
    Queue<ThreadTask> syncBuffer = tasks.Switch();  // 把生产端正在写的队列和消费端要读的队列整体交换
    while (syncBuffer.Count > 0)
        syncBuffer.Dequeue().Execute();
}

tasks.Switch() 的语义是：把队列内部"正在被生产者入队"的那一半和"准备给消费者读"的那一半整体对调，返回消费者这一半。这样消费者逐个 Dequeue 执行时，生产者仍然可以往新的"入队半边"写，互不阻塞。比起每取一个都加锁，切换一次开销小得多，适合"批量入队、批量消费"的场景。

两个队列分别在两个线程被消费：

// 主线程 Update 里（UniversalScene.cs:307）消费"回主线程"队列
public virtual void Update()
{
    RefreshThreadTask(m_UnityExecuteSyncBuffer);   // 跑 ExecuteInUnity 丢进来的任务
    ...
}

// 工作线程 ThreadUpdate 里（UniversalScene.cs:272）消费"去后台"队列
protected void ThreadUpdate()
{
    RefreshThreadTask(m_ThreadExecuteSyncBuffer);  // 跑 ExecuteInThread 丢进来的任务
    ...
}

所以一个任务到底在哪个线程跑，完全取决于你把它塞进哪个队列。

4.6 ThreadTask 是什么

ThreadTask 是 UniversalScene 内部定义的 struct（行 444），本质是"一个参数 + 一个 Action<object>"的包装：

public struct ThreadTask
{
    private Action<object> m_Action;
    private object m_Param;

    public void Execute() { m_Action(m_Param); }

    public ThreadTask(object param, Action<object> action) { ... }       // (param, action)
    public ThreadTask(Action<object> action, params object[] param) { ... } // (action, params)
}

两个构造重载允许 (param, action) 或 (action, params) 两种写法。用 struct 而不是 class 是为了避免每个任务一次堆分配，在线程高频交互场景下减少 GC 压力。

4.7 真实用法：嵌套切线程

最典型的用法是"丢后台算 → 算完切回主线程处理结果"的嵌套。来看一个真实在用的例子，MCManager 加载完一个 patch 后处理网络数据（Assets/Scripts/Core/MCCore/Scene/MCManager.cs:181）：

public virtual void OnPatchLoad(TiledPatch patch, Action<TiledPatch> callback)
{
    MCPatch mcPatch = (MCPatch)patch;
    mcPatch._asyncCombiner.AddCompletionCall(() =>
    {
        Scene.ExecuteInThread(new UniversalScene.ThreadTask(patch, (obj) =>     // ① 丢到后台线程
        {
            ((MCPatch)obj).ProcessNetData();                                    //   后台：解析网络数据（纯 CPU，不碰 Unity）

            Scene.ExecuteInUnity(new UniversalScene.ThreadTask(obj, (obj2) =>   // ② 再丢回主线程
            {
                callback((MCPatch)obj2);                                        //   主线程：回调上层、更新加载状态
                ((ILoadingState)Scene).RemoveLoadingTilePatch(((MCPatch)obj2).UV.ID);
            }));
        }));
    });
    ...
}

这里的嵌套结构是关键：

主线程调用 OnPatchLoad
        │
        │  AsyncCombiner 完成回调（可能在主线程）
        ▼
┌─────────────────────────────────────────────┐
│ ExecuteInThread( 任务A )                     │   入 m_ThreadExecuteSyncBuffer
└─────────────────────────────────────────────┘
        │
        ▼  （稍后）工作线程 ThreadUpdate 取出任务A 执行
┌─────────────────────────────────────────────┐
│ ProcessNetData()    ← 后台线程，纯数据计算    │
│                                             │
│   ExecuteInUnity( 任务B )   入 m_UnityExecuteSyncBuffer  │
└─────────────────────────────────────────────┘
        │
        ▼  （下一帧）主线程 Update 取出任务B 执行
┌─────────────────────────────────────────────┐
│ callback(patch)     ← 主线程，可碰 Unity API  │
│ RemoveLoadingTilePatch(...)   更新加载状态    │
└─────────────────────────────────────────────┘

这样就把"耗 CPU 但不碰 Unity"的活留在后台、"要碰 Unity 或要改加载状态"的活切回主线程，两边各司其职。

另一个常见的真实用法是"批量修改某个组件的内部状态"。比如 SceneObjectManager.AddOcclusionMask（Assets/Scripts/Engine/UniversalScene/Components/SceneObjects/SceneObjectManager.cs:101）——主线程调用，但实际往 OcclusionMasks 列表里加数据的动作丢到后台线程去做，因为这个列表在工作线程也会读，统一在后台线程改避免冲突：

public void AddOcclusionMask(SceneEntityOcclusionMask mask)
{
    Scene.ExecuteInThread(new UniversalScene.ThreadTask((null), (obj) =>
    {
        OcclusionMasks.Add(mask);
    }));
}

4.8 线程安全注意点

这套机制定下了一条隐含的分工红线：

类型可以在哪个线程做例子
Unity API（创建 GameObject、改 Transform、挂组件、Debug.Log 大多数情况）必须主线程ExecuteInUnity 丢回主线程后的 callback
纯数据计算（解压、网格构建、四叉树插入、坐标换算）后台线程ExecuteInThread 任务体、IThreadRunnable.ThreadUpdate
改加载状态（SetLoadingState / RemoveLoadingTilePatch 等）可在后台线程（用 ConcurrentDictionary + lock 保证安全）MCScene 的 m_LoadingChunks / m_LoadingTilePatchs 都是 ConcurrentDictionary
读组件共享字段看具体情况，跨线程读写必须用 volatile / lock / 并发容器m_Location 用 volatile 保护，m_IsThreadRunning/Exit 同理
易错点：不是所有"看起来像数据"的操作都能丢后台。任何会触发 Unity 内部状态变更的调用（哪怕是间接的）都必须在主线程。写新功能时，拿不准就一律走 ExecuteInUnity，宁可慢一点也不要崩。

5. SetLocation —— 驱动重加载的总开关（重点）

5.1 它是什么

SetLocation 是 UniversalScene 的 virtual 方法，MCScene 重写了它。它是整个大世界"中心点移动 → 触发重新加载"的入口。基类版本很朴素（行 114）：

public virtual void SetLocation(YKVector3d location, Action callBack = null)
{
    m_Location = location;
}

基类只更新坐标，不触发任何加载。真正有逻辑的是 MCScene.SetLocation（Assets/Scripts/Core/MCCore/Scene/MCScene.cs:849）。

5.2 MCScene.SetLocation 的完整流程

先把代码贴出来再拆解：

// Assets/Scripts/Core/MCCore/Scene/MCScene.cs:849
public override void SetLocation(YKVector3d location, Action callBack = null)
{
    if (IsSceneInLoading && location != m_Location && m_Location != new YKVector3d(Double.MinValue, ...))
        Log.Debug("Scene In Loading but SetLocation be call| ...");

    if (m_SetLocationCallBack == null)                       // 当前没有正在跑的 SetLocation
    {
        double dis = (location - m_Location).Magnitude;      // 新中心和老中心的距离
        m_Location = location;
        if (callBack != null)
        {
            if (dis > 32)                                   // ① 距离大于 32，触发重加载
            {
                m_SetLocationCallBack = callBack;
                FinishLoadingCondition = SceneLoadedCondition.NearbyLod0_Finish;
                SwitchToLoadingState(true);                  // 进入"加载中"状态，提高线程优先级
                Application.backgroundLoadingPriority = ThreadPriority.Normal;
                QualitySettings.asyncUploadBufferSize = 16;
                QualitySettings.asyncUploadTimeSlice = 2;

                ExecuteInThread(new UniversalScene.ThreadTask(null, (obj) =>   // ② 丢后台线程
                {
                    if (this is ILoadingState loadingState)
                        loadingState.ResetLoadingState();                     //   清空加载状态机

                    SetOnLoadedAction(() =>                                    // ③ 注册"加载完成"回调
                    {
                        Vector4 pos = new Vector4((float)m_Location.x - 5000, ...);
                        UniversalRenderPipeline.asset.environmentData.RefetchGroundVegeAtPos(pos); // 更新草地

                        if (m_HasWaitSetLocation)                              // 有排队的 SetLocation，接着跑
                            SetLocation(m_WaitLocation, m_WaitSetLocationCallBack);

                        m_WaitSetLocationCallBack = null;
                        m_HasWaitSetLocation = false;

                        m_SetLocationCallBack?.Invoke();                       // 通知本次的 callBack
                        m_SetLocationCallBack = null;
                    });
                }));
            }
            else                                             // ④ 距离 ≤ 32，场景无变化，直接回调
            {
                callBack?.Invoke();
            }
        }
    }
    else                                                     // 当前已有 SetLocation 在跑，这次请求排队
    {
        m_WaitLocation = location;
        m_WaitSetLocationCallBack = callBack;
        m_HasWaitSetLocation = true;
    }
}

5.4 关键设计点逐条说明

阈值 32 是什么意思

dis > 32 就是重加载阈值，单位是 block（方块）坐标距离。意思是：新中心和老中心移动距离超过 32 个方块，才认为"玩家走到了一个新区域"，需要重新扫描加载周围的 chunk 和物件；否则场景内容基本没变，直接回调啥也不干。这个阈值避免了原地小碎步移动反复触发重加载。

核实说明：前期探查提到"距离 > 32 触发重加载"，代码确认无误。但要注意阈值是 > 32（严格大于），不是 >=。距离正好 32 时不触发，走"直接回调"分支。

为什么用 ExecuteInThread 包 ResetLoadingState

ResetLoadingState 会清空 m_LoadingState 标记和 m_LoadingTilePatchs / m_LoadingChunks / LoadingEntitys 这些集合（Assets/Scripts/Core/MCCore/Scene/MCScene.cs:465）。这些容器在工作线程也会读写，所以清空动作也得放到工作线程去做，和后台扫描逻辑在同一线程，避免一边清一边写的竞争。这就是为什么明明 SetLocation 是主线程调进来的，ResetLoadingState 却要绕一圈 ExecuteInThread。

SetOnLoadedAction 注册的完成回调

SetOnLoadedAction（行 479）把一个 Action 存到 OnLoadedAction 字段。等整个场景加载状态机跑完（LoadingState == All），OnSceneLoadingFinish（行 912）会调 Loaded()，Loaded()（行 693）就触发这个 OnLoadedAction。所以这个回调本质是"本次重加载全部完成"的通知。

回调里三件事：更新草地（因为中心移动了，草地采样区要跟着挪）、处理排队的下一个 SetLocation、调本次的 callBack。

并发请求排队

场景重加载是个异步过程，没跑完时如果又来一个 SetLocation，不能直接拒，也不能并发跑两个（状态机会乱）。所以用一个单槽缓冲：把后来的请求存到 m_WaitLocation / m_WaitSetLocationCallBack，标记 m_HasWaitSetLocation = true。等当前重加载完成、OnLoaded 回调触发时，检查这个标记，如果有排队就把排队的 SetLocation 递归调一次。也就是说同一时刻最多有一个 SetLocation 在跑，多余的排成一个长度为 1 的队列。

5.5 谁在喂坐标给 SetLocation

这是"谁在驱动大世界中心移动"的答案。答案在 WorldSceneSystem（Assets/Scripts/Game/GameWorld/Game/OpenWorldGameSystem/WorldSceneSystem.cs:111）。它的 Update 每帧调 SetScanCenterLocation：

{
    base.Update();
    if (IsInitScene)
    {
        SetScanCenterLocation();   // 每帧把"扫描中心"喂给 MCScene.SetLocation

// WorldSceneSystem.cs:270
public void SetScanCenterLocation()
{
    switch (m_ScanType)
    {
        case ScanType.Role:
            if (m_MainPlayer != null)
                MCScene.SetLocation(m_MainPlayer.Position);      // 跟主角
            break;
        case ScanType.Camera:
            MCScene.SetLocation(m_CameraTransform.position.ToVector3d()); // 跟相机
            break;
        case ScanType.Location:
            MCScene.SetLocation(m_ScanLocaton);                  // 跟固定坐标
            break;
    }
}

ScanType 是个枚举（行 26），三种模式：

ScanType值扫描中心取自用在哪
Role1主角 m_MainPlayer.Position正常游戏，主角已入场。加载场景完成、主角入场后 MainPlayerJoin 调 ChangeScanTypeToRole 切到这个模式（行 219）
Camera2场景相机位置默认值（m_ScanType = ScanType.Camera，行 57）。特殊场景跟相机走
Location3固定坐标 m_ScanLocaton刚登录还没主角时。LoadScene（行 171）会先 ChangeScanTypeToLocation(px,py,pz) 把服务器给的主角坐标先当扫描中心，等场景加载完、主角真的进来了再切 Role

所以整体节奏是：登录 → 用服务器给的主角坐标当 Location 模式加载场景 → 主角入场后切 Role 模式，之后每帧跟主角。中间如果某些玩法需要跟相机，可以 ChangeScanType(ScanType.Camera) 切过去。

5.6 关于 JumpToPosition 里的注释用法

前期线索提到 WorldSceneSystem.JumpToPosition 里有 ExecuteInThread → SetLocation → ExecuteInUnity 的嵌套用法。核实后确认：整段逻辑当前被 /* ... */ 注释掉了，方法直接 return false（Assets/Scripts/Game/GameWorld/Game/OpenWorldGameSystem/WorldSceneSystem.cs:314-365）。也就是说这个跳转分支目前未启用。它的写法确实能说明"后台改坐标 → 主线程挪主角"的思路，但作为真实在用的例子，不如第 4.7 节 MCManager.OnPatchLoad 那个来得可靠。本篇以真实在用的为准。

6. UniversalSceneManager —— 场景的创建与管理

UniversalSceneManager 是一个静态工具类（static class，Assets/Scripts/Engine/UniversalScene/UniversalSceneManager.cs:11），namespace 同样是 UnityEngine.Scene.Universal。它管三样东西：

private static GameObject m_SceneRoot;                                          // 所有场景的根节点
private static Dictionary<string, UniversalScene> m_Scenes = new Dictionary<...>(); // 所有活着的场景，key 是 scenePath
private static UniversalScene m_MainScene;                                      // 当前主场景

6.1 CreateScene<T> —— 创建一个带配置的场景

这是正式加载大世界用的方法（行 380）：

public static T CreateScene<T>(string scenePath, Action<T> callback,
                                bool autoLoad = true, params object[] args)
    where T : UniversalScene

步骤：

  1. 确保场景根节点 m_SceneRoot 存在（没有就 new GameObject("UniversalSceneRoot")，编辑器模式下还挂个 ApplicationQuit 组件用于退出时清理）。
  2. 查字典，如果这个 scenePath 已经在场景里了，报错并返回已有那个。
  3. 否则 new GameObject 挂到 m_SceneRoot 下，AddComponent<T>() 把场景类作为 MonoBehaviour 挂上去。
  4. 设 MainScene = t、t.ScenePath = scenePath，加进字典。
  5. 调 t.Initialize(args)（场景的初始化，注册组件就在这里）。
  6. autoLoad = true 时调 t.OnLoad(() => callback(t))，加载完成后回调外部；autoLoad = false 直接回调，让外部自己控制加载时机。

真实调用点在 WorldSceneSystem.LoadScene（Assets/Scripts/Game/GameWorld/Game/OpenWorldGameSystem/WorldSceneSystem.cs:176）：

public void LoadScene(string path, float px, float py, float pz, bool isNet = true)
{
    ChangeScanTypeToLocation(px, py, pz);
    bool isNetManager = isNet;
    m_McScene = UniversalSceneManager.CreateScene<MCScene>(path, scene => { OnLoadingFinishAction(); }, true, isNetManager);
    MCScene.LoadingType = LoadingType.Login;
    m_MCNetManager = m_McScene.MCManager;
    IsInitScene = true;
    MCScene.SetLodInfo(m_LodInfo);
    SetScanCenterLocation();
    ...
}

注意第四个参数 isNetManager 通过 args 传进 MCScene.Initialize，Initialize 里根据它决定注册 MCNetworkManager（联机）还是 MCManager（单机/编辑器）还是 MCModifyManager（修改器模式），见 Assets/Scripts/Core/MCCore/Scene/MCScene.cs:37-52。同一个 MCScene 类，靠这个参数切出三种不同的"MC 管理器"实现。

6.2 CreateEmptyScene<T> —— 创建空场景

不需要从磁盘加载配置、直接给个默认空配置的场景（行 50）。和 CreateScene 的区别是：它不传 scenePath，用 CreateEmptySceneConfig() 造一份默认 YKSceneConfig（含一组默认 ScanGroup，行 124 那段内置 JSON），用场景名（EmptyScene_{hash}）作字典 key，并且 Initialize(false)（非网络模式）+ 立即 OnLoad(null)。用在不需要真实地形数据的场合。

6.4 ClearAll —— 关闭时清理一切

游戏关闭或切场景时调（行 471）：WorldSceneSystem.Shutdown 里就会调它（Assets/Scripts/Game/GameWorld/Game/OpenWorldGameSystem/WorldSceneSystem.cs:137）：

public static void ClearAll()
{
    if (m_SceneRoot != null)
        UnityEngine.Object.Destroy(m_SceneRoot.gameObject);   // 销毁场景根节点（连带所有子场景 GameObject）
    m_SceneRoot = null;
    foreach (var temp in m_Scenes)
        temp.Value.UnLoad();                                  // 逐个场景走 UnLoad（停线程、销毁组件、清资源）
    m_Scenes.Clear();
    m_MainScene = null;
    SceneAssemblerManager.Instance.ClearAll();                // 清理各种全局 Manager
    RegionMapManager.Instance.Release();
    VFXEffectManager.Clear();
    GraphInteractionObjectPool.Clear();
    PostProcessDrawManager.Instance.Dispose();
}

注意它既 Destroy(m_SceneRoot) 又遍历 UnLoad，是因为 UnLoad 里除了销毁 GameObject 还做了停线程、清组件、释放资源等善后，光靠 Unity 销毁 GameObject 不会触发这些（UniversalScene 没有把清理放在 OnDestroy 里，而是放在公开的 UnLoad() 方法里，需要主动调）。

架构提示：UniversalSceneManager 只管"场景实例的生与死 + 字典索引"，不碰场景内部逻辑。场景内部能力由 SceneComponent 拼装，场景的加载/卸载细节由 UniversalScene/MCScene 自己负责。这是个职责很干净的拆分。

7. SceneObjectManager —— 场景物件的总管（定位介绍）

它是 UniversalScene 上负责"物件"的组件，本篇只做定位，细节留到第 6 篇展开。

定义在 Assets/Scripts/Engine/UniversalScene/Components/SceneObjects/SceneObjectManager.cs:10：

public class SceneObjectManager : SceneComponent, IThreadRunnable, ILoadingStateSwitch

它是个 SceneComponent（所以挂场景上、跟场景生命周期绑定），同时实现了 IThreadRunnable（有后台 ThreadUpdate，参与工作线程的四叉树维护等）和 ILoadingStateSwitch（加载状态切换时它会跟着调整）。

它的职责是管理场景里所有 SceneEntity（场景物件）的组织方式，包括：

  • LOD 分组：把物件按 LOD 级别分组管理（SceneObjectLodGroup[]）。
  • 四叉树：用四叉树按空间位置组织物件，方便按区域查询（m_AllQuadTreeEntity）。
  • 激活集合：维护当前帧"激活/可见"的物件列表 m_ActiveEntitys（一个 DynamicArray<SceneEntity>）。
  • 遮挡 mask：维护遮挡剔除用的 mask 列表 OcclusionMasks。

对外最常用的两个入口：

public void UpdateLod(float baseVPHeight, Vector3 cameraPos)

// 行 521：初始化
public void Init()

UpdateLod 由 MCScene.UpdateLod 调用（Assets/Scripts/Core/MCCore/Scene/MCScene.cs:262），每帧根据 LodInfo（相机 fov/aspect/position）算出 baseVPHeight 喂进来。这就是物件 LOD 跟相机走的入口。

MCScene.SceneObjectManager 这个属性（Assets/Scripts/Engine/UniversalScene/UniversalScene.cs:120）就是它，外部（包括 WorldSceneSystem 的大量封装接口）都通过 MCScene.SceneObjectManager 来访问物件层能力。物件的四叉树、LOD 分组、激活集合、mask 具体怎么运作，是第 6 篇的主题。

8. 小结与下一站

本篇要点回顾

  1. UniversalScene 是 MonoBehaviour 场景基类，把"可加载卸载、自带工作线程、组件化、LOD 更新、加载状态机、扫描中心"这些通用能力下沉到一层，让 MCScene 专注体素逻辑。
  2. SceneComponent 是自建的轻量组件体系（非 MonoBehaviour），用 Dictionary<Type, SceneComponent> 管理，RegisterComponent<T> 反射创建、GetComponent<T> 查表，生命周期由场景转发。
  3. 工作线程是核心设计：ExecuteInThread 入后台队列、ExecuteInUnity 入主线程队列，双缓冲 ConcurrentQueue.Switch() 交换消费。耗 CPU 不碰 Unity 的活丢后台、要碰 Unity 的活切回主线程，可嵌套。
  4. SetLocation 是重加载总开关：MCScene 重写，dis > 32 触发重加载（SwitchToLoadingState(true) + 后台 ResetLoadingState + 注册 OnLoaded 回调），否则直接回调。并发请求单槽排队。WorldSceneSystem.SetScanCenterLocation 每帧按 ScanType（Role/Camera/Location）喂坐标。
  5. UniversalSceneManager 是静态工具类，管 SceneRoot、场景字典、MainScene，提供 CreateScene<T> / CreateEmptyScene<T> / ClearAll 等。
  6. SceneObjectManager 是物件总管组件，管 LOD 分组、四叉树、激活集合、遮挡 mask，细节第 6 篇展开。

承上启下

这一层搭好了"场景骨架 + 工作线程 + 组件 + 重加载驱动"。下一篇我们走进 MCScene 真正的体素世界内部，讲它的核心数据结构——MCWorld、MCChunk、MCBlock 这些概念是什么、怎么用整数坐标和 PID 编码体素位置。那是体素大世界的"数据底座"，和本篇的"场景底座"一起构成整个系统的两层地基。
