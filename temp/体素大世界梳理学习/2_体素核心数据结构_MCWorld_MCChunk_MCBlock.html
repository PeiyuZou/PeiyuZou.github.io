1. 存储模型总览

体素大世界本质上就是一个"用 3D 网格存方块"的世界。整个世界被切成一个巨大的立方体网格，每个网格单元存一个方块（block）。方块数量极大，不可能用一个无限大的三维数组铺满整个世界，所以项目用的是一个三层嵌套的存储模型：

世界 (MCManager)
   │
   │  持有 5 个 MCWorld（5 层 LOD，每层 block 实际大小不同）
   │  ← 本篇只讲"单个 MCWorld 内部怎么存"，5 层为什么这么设计在第 3 篇
   ▼
单层 MCWorld
   │
   │  持有 Dictionary<long, MCChunk> Chunks
   │  按 chunkIndex(long) 索引每个 chunk，稀疏存放（只有玩家附近的 chunk 才存在）
   ▼
MCChunk（一个 32×32×32 的方块组）
   │
   │  持有 MCBlock[] Blocks，长度 = 32³ = 32768
   │  chunk 实际覆盖的世界边长 = BlockSize × ChunkSize
   ▼
MCBlock（一个方块，struct，6 字节）
     Type(ushort) + BlockMask(ushort) + CuttingType(byte)

三层的包含关系是：世界 → 多层 MCWorld → 每层一堆 MCChunk → 每个 chunk 是 32³ 个 MCBlock。从大到小，每一层都用不同的数据结构，各自解决一个核心问题：

层数据结构解决什么问题
MCBlockstruct（值类型）方块数量巨大，要内存紧凑、无 GC、缓存友好
MCChunkMCBlock[32768] 一维数组把 32³ 个方块连续摆放，用"坐标 → 数组下标"换算访问
MCWorldDictionary<long, MCChunk>大世界稀疏且动态加载，只有玩家附近的 chunk 才存在
一句话定位：本篇讲的是"单层 MCWorld 内部怎么存方块"。MCManager 持有 5 个 MCWorld，每层 block 实际大小不同（1/2/4/8/16），但单个 MCWorld 的存储结构是一样的。5 层为什么这么设计、怎么调度，留到第 3 篇展开。

这一层的全部代码集中在一个文件里：Packages/com.unity.mccore/Runtime/MCWorld.cs（namespace MCCore）。MCWorld、MCChunk、MCBlock、MCBlockPool 四个类型都挤在这一个文件，因为它们是一个不可分割的"存储四件套"。下面从最小的 MCBlock 开始往上讲。

2. MCBlock —— 一个方块是什么

MCBlock 是体素世界的最小单元，定义在 Packages/com.unity.mccore/Runtime/MCWorld.cs:388：

[Serializable, StructLayout(LayoutKind.Explicit, Pack = 1)]
public struct MCBlock
{
    public static MCBlock Empty { get; }

    public const ushort BREAKABLE_MASK = 0x1;        // 是否可破坏
    public const ushort CROSS_MASK = 0x2;            // 是否可穿越（用于开关门）
    public const ushort GRAPHICS_CLOSE_MASK = 0x4;   // 是否关闭渲染
    public const ushort COLOR_MASK = 0b_11111111000; // 颜色站位区间

    [FieldOffset(0)] public ushort Type;         // 类型，ArtId（美术 Id）。Type == 0 表示空块
    [FieldOffset(2)] public ushort BlockMask;    // 格子属性，位标志组合
    [FieldOffset(4)] public byte   CuttingType;  // 切割类型（三角形朝向等）

    public int  Color           => (BlockMask & COLOR_MASK) >> 3;
    public bool IsClosedGraphics => (BlockMask & GRAPHICS_CLOSE_MASK) != 0;
    public bool IsCross         => Type == 0 || (BlockMask & CROSS_MASK) != 0;
}

2.1 三个字段各是什么

字段类型偏移含义
Typeushort0方块的美术 Id（ArtId），决定渲染成什么样子。Type == 0 表示这是一个空块（注释原文："Type '0' is inactivated block"）。上层判断空块用 MCObjectUtils.IsEmptyBlock(type)（Assets/Scripts/Game/GameUtils/MCObjectUtils.cs:167），运行时就是判 type == 0。
BlockMaskushort2方块属性位标志，把"可破坏 / 可穿越 / 关闭渲染 / 颜色"挤在一个 ushort 里。注释里的布局是 {[5]预留 [8]颜色 [1]可破坏 [1]可穿越 [1]关闭渲染}。改它必须用位运算（& |），不能直接 = 赋值，否则会覆盖其它位。
CuttingTypebyte4切割类型，描述非完整方块的形状。两种情况：三角形块旋转方向（&3 得 0↓/1→/2↑/3←），或正方形块切割方向（23↓/20←/21↑/22→）。具体是哪种由 ObjectInfoConfig 配置决定。

三个字段加起来 6 字节。文件顶部有一行明确警告（MCWorld.cs:386）："增减结构体成员变量，或者改变成员的 Type，请注意结构体内存布局排紧"。也就是说这 6 字节的布局是刻意压紧的，改字段类型或顺序要同步检查内存布局。

2.2 为什么用 struct 不用 class（重点）

这是体素世界最底层的一个关键设计决策。一个 chunk 就有 32³ = 32768 个方块，玩家周围同时存在几十上百个 chunk，方块总量轻易上百万。如果 MCBlock 是 class：

  • 每个方块都是堆上的一个对象，百万级对象光是引用就占几十 MB。
  • 每个对象有对象头开销（类型指针、同步块），实际数据 6 字节但对象本身可能占 16+ 字节。
  • 对象分散在堆各处，访问时缓存命中差——遍历一个 chunk 的方块要反复跳内存地址。
  • 创建/销毁产生大量 GC 压力。

改成 struct 之后，MCBlock[] Blocks 就是一段连续的、紧凑的值类型内存：32768 个方块紧紧挨着，每个 6 字节，整段数组约 192 KB。遍历时 CPU 缓存能成片地预取，访问相邻方块几乎不掉缓存；赋值是值拷贝，没有对象分配，没有 GC。这就是"用 struct 存方块"的全部意图——让海量方块在内存里连续、紧凑、零 GC。

2.3 StructLayout 显式布局

[StructLayout(LayoutKind.Explicit, Pack = 1)] 配合每个字段的 [FieldOffset(n)]，是手动指定每个字段在 struct 里的精确字节偏移：

偏移:   0      1      2      3      4      5
       ┌──────────────┬──────────────┬──────┐
       │  Type(ushort)│ BlockMask    │Cut   │
       │              │  (ushort)    │(byte)│
       └──────────────┴──────────────┴──────┘
       └───────────── 6 字节，Pack=1 不留空隙 ─────────────┘

Pack = 1 让字段间不按默认对齐留空隙（ushort 默认会 2 字节对齐，这里强制 1 字节对齐）。LayoutKind.Explicit 还允许字段重叠（虽然这里没用上重叠，但留了将来把多个小字段压进同一段空间的可能）。这套写法的目的就是把 struct 压到最小、布局完全可控，让百万方块的内存占用可预测。

2.4 MCBlock.Empty 是什么

public static MCBlock Empty { get; } 是一个静态只读属性，返回一个所有字段都是默认值（Type=0、BlockMask=0、CuttingType=0）的 MCBlock。因为 struct 默认值就是全零，这里相当于 default(MCBlock)。它的作用是给查询失败时一个统一的"空块"返回值——比如 MCWorld.TryGetBlock 找不到 chunk 时就返回 MCBlock.Empty（见 MCWorld.cs:69 和 :82），调用方拿到 Type==0 就知道"这里没方块"。

特殊类型块：Type 还有一批高位预留值（MCBlockTypeLEDefine，MCWorld.cs:353）：AIRWALLBLOCK(65401) 空气墙、VOIDBLOCK(65501)/DVOIDBLOCK(65502) 非空占位块。编辑器下 IsEmptyBlock 会把 VOID 也当空块对待，运行时只认 Type==0。这些是给关卡编辑器用的特殊占位，正常游戏方块走 1~65500 的正常 ArtId 区间。

2. MCChunk —— 一个 32³ 的方块组

MCChunk 把 32768 个方块打包成一组，定义在 Packages/com.unity.mccore/Runtime/MCWorld.cs:575。它是个 class，实现 IDisposable：

public class MCChunk : IDisposable
{
    public readonly MCBlock[] Blocks;   // 长度 32³ = 32768

    public YKVector3f Offset;   // chunk 在世界中的左下角坐标
    public YKVector3i Index;    // chunk 的整数索引（三个分量可正可负）
    public byte BlockSize;      // 每个 block 的实际边长（1/2/4/8/16）
    public byte ChunkSize;      // 每边 32 个 block（固定 32）

    public byte IsDirty = 0;    // 脏标记，位标志 MCChunkDirtyFlags
    // ...
}

几个字段的几何关系要理清：Offset 是这个 chunk 在世界坐标系里的左下角坐标（float3）；Index 是 chunk 的整数编号（用来压成 long 当字典 key，下一节讲）；BlockSize 是单个方块的实际边长，ChunkSize 是每边多少个方块（恒为 32）。所以一个 chunk 实际覆盖的世界边长是：

chunk 世界边长 = BlockSize × ChunkSize

  BlockSize = 1  → chunk 边长 32    （LOD0，最细）
  BlockSize = 2  → chunk 边长 64    （LOD1）
  BlockSize = 4  → chunk 边长 128   （LOD2）
  BlockSize = 8  → chunk 边长 256   （LOD3）
  BlockSize = 16 → chunk 边长 512   （LOD4，最粗）

注意 BlockSize 和 ChunkSize 都被特意声明成 byte 而不是 int/float。代码注释（MCWorld.cs:584-585）写明原因：ChunkEntity 常驻内存大概有 17 万个，积少成多，把这两个小整数压成 byte 能省下不少内存。Offset 也从 double 改成了 float（MCWorld.cs:579 注释"从 double 改为 float，减少 4 字节分配"）。这些都是百万级对象场景下的刻意抠内存。

3.1 坐标 ↔ 数组下标换算（重点）

这是 MCChunk 最核心的方法。一个世界坐标怎么落到这个 chunk 里的某个方块上？答案在 PositionToArrayIndex（MCWorld.cs:712）：

public int PositionToArrayIndex(float worldX, float worldY, float worldZ)
{
    float blockSizeInverse = 1.0f / BlockSize;
    int x = (int)Math.Floor((worldX - Offset.x) * blockSizeInverse);
    int y = (int)Math.Floor((worldY - Offset.y) * blockSizeInverse);
    int z = (int)Math.Floor((worldZ - Offset.z) * blockSizeInverse);

    return (x * ChunkSize + y) * ChunkSize + z;
}

拆成两步：

  1. 世界坐标 → chunk 内 block 索引 (x, y, z)：先减去 chunk 左下角 Offset，得到"在这个 chunk 内的相对坐标"；再除以 BlockSize（乘倒数避免除法），向下取整，得到三个 0~31 的整数索引。Math.Floor 保证负坐标也正确（比如相对坐标 -0.5 会取到 -1，对应越界，由上层保证不进来）。
  2. 三维索引 (x,y,z) → 一维数组下标：(x * ChunkSize + y) * ChunkSize + z。这是一个标准的"行主序"展开，把 32×32×32 的三维空间铺平成 0~32767 的一维下标。

下标展开公式的几何含义：

一维下标 = (x * 32 + y) * 32 + z
         = x * 1024 + y * 32 + z

  x 走一格  → 下标跳 1024（一整层 yz 平面，32×32=1024 个）
  y 走一格  → 下标跳 32   （一整行 z，32 个）
  z 走一格  → 下标跳 1    （连续相邻）

  所以 Blocks 数组在内存里是按 "x 最外层、y 中层、z 最内层" 排列的。
  沿 z 方向遍历是连续内存访问，缓存最友好。

MCChunk 还提供了一对索引器配合这个下标（MCWorld.cs:777-791）：this[YKVector3i index]、this[int x,y,z]、this[int index]。其中 this[int x,y,z] 和 this[int index] 返回的是 ref MCBlock——按引用返回，这样上层可以直接 chunk[x,y,z] = newBlock 原地改写方块，不用拆开 struct 再拷回去。这是 struct 方块能被修改的关键。

3.2 脏标记设计

MCChunk 用一个 byte IsDirty 字段配合一组位标志（MCChunkDirtyFlags，MCWorld.cs:557）记录"这个 chunk 哪里变了，需要上层处理"：

标志值含义
ChunkBlockDirty0x01方块数据脏——方块类型/属性变了，mesh 需要重建
ChunkEntityDirty0x02实体数据脏——异形块/装饰物变了，entity 需要刷新
ChunkAllDirty0x03全脏，同时标记方块和实体

对应一组标记/清除方法：MarkBlockDirty() / MarkEntityDirty() / MarkAllDirty() / ClearBlockDirty() / ClearEntityDirty() / ClearAllDirty()，以及两个只读判断 IsBlockDirty / IsEntityDirty。实现都是位运算（IsDirty |= flag、IsDirty & ~flag）。

本篇只讲标记本身是什么、谁写它：

  • 方块被修改时调 MarkBlockDirty()，告诉上层"这个 chunk 的方块数据变了，mesh 要重建"。
  • 异形块/装饰物变化时调 MarkEntityDirty()，告诉上层"这个 chunk 的实体列表要刷新"。
  • chunk 首次创建时直接置全脏：IsDirty = MCChunkDirtyFlags.ChunkAllDirty（MCWorld.cs:665），确保首次渲染会执行所有构建逻辑。

上层怎么响应这些脏标记（重建 mesh、刷新 entity）是第 4 篇的主题，本篇不展开。这里只需记住：脏标记是 chunk 向上层传递"我需要被处理"的信号，用位标志把"方块脏"和"实体脏"两条独立通道挤进一个 byte。

3.3 MCBlock[] 怎么来、怎么回池 —— MCBlockPool

chunk 的 Blocks 数组长度固定 32768，会随着 chunk 反复创建/销毁而频繁分配回收。直接 new MCBlock[32768] 的问题是：数组本身是大对象，频繁分配压 GC，而且新数组内容是全零（即全空块），如果复用旧数组还能省一次清零。所以项目搞了 MCBlockPool（MCWorld.cs:458）专门池化这种 32768 长度的数组：

public static class MCBlockPool
{
    static Queue<MCBlock[]> BlocksPool = new Queue<MCBlock[]>(1024);
    public static int BlockLength = 32 * 32 * 32;   // 32768

    public static void InitMCBlockPool(int poolSize = 512);  // 预分配 poolSize 个数组入池
    public static MCBlock[] AllocateMCBlocks();              // 取：池里有就出队，没有才 new
    public static void RecycleMCBlocks(MCBlock[] Blocks);    // 还：Array.Clear 清零后入队
}

设计意图很直接：用对象池复用 32768 长度的数组，避免反复分配。chunk 构造时 Blocks = MCBlockPool.AllocateMCBlocks()（MCWorld.cs:654），chunk 销毁时 Dispose() 把数组 RecycleMCBlocks 还回池里（MCWorld.cs:689）。回收时会 Array.Clear 把数组清成全空块，下次拿出来直接就是干净状态。

池大小由画质决定：MCManager 构造时调 MCBlockPool.InitMCBlockPool(512)，低画质档再乘 0.5（Assets/Scripts/Core/MCCore/Scene/MCManager.cs:128-135）。有个全局开关 MCWorld.EnableMCBlockPool = true（MCWorld.cs:17），关掉就退回直接 new。池操作都加了 lock(_synLock)，因为 chunk 可能在工作线程创建/销毁。

注意：MCChunk 有个析构函数 ~MCChunk() { Dispose(); }（MCWorld.cs:695），兜底防止 chunk 没被显式 Dispose 时数组没回池。但析构时机不可控，正常流程都应显式调 Dispose。另外构造函数有个重载可以外部传入 MCBlock[] blocks，这种情况 _isDisposed = true（MCWorld.cs:678），Dispose 时不会把这个外部数组回池——因为这块内存不属于池。

4. MCWorld —— 一层 LOD 的 chunk 容器

MCWorld 是一层 LOD 里所有 chunk 的容器，定义在 Packages/com.unity.mccore/Runtime/MCWorld.cs:8：

public class MCWorld
{
    public readonly float BlockSize = 1;     // 每个 block 的实际边长
    public readonly int ChunkSize = 32;      // 每边 32 个 block

    // 性能优化：预算好的乘积和倒数，避免每次查询都算
    private readonly float ChunkSizeMulBlockSize;        // = ChunkSize * BlockSize
    private readonly float ChunkSizeMulBlockSizeInverse; // = 1 / (ChunkSize * BlockSize)

    public static readonly bool EnableMCBlockPool = true;

    public Dictionary<long, MCChunk> Chunks = new Dictionary<long, MCChunk>();

    public MCWorld(float blockSize, int chunkSize) { ... }
}

核心就是一个 Dictionary<long, MCChunk> Chunks，按 chunkIndex（一个 long）存 chunk。两个只读属性 BlockSize 和 ChunkSize 决定这一层的几何尺度。构造时还预算了两个派生量 ChunkSizeMulBlockSize（一个 chunk 的世界边长）和它的倒数，存起来给后面的坐标换算用，避免每次查询都重复算乘除——这也是百万级查询场景下的小优化。

4.1 为什么用 Dictionary<long, MCChunk> 而不是 3D 数组（重点）

这是 MCWorld 这一层最关键的设计决策。最直观的存法是开一个三维数组 MCChunk[,,] 按 chunk 的 (x,y,z) 索引存。但体素大世界有两个特性让 3D 数组不可行：

  • 世界极大：地图尺度动辄几万甚至几十万 block，对应的 chunk 数量在三个方向上都是上千甚至上万，三维数组要 10^9 量级的槽位，根本开不出来。
  • 稀疏且动态：玩家某一时刻只站在世界的一个点，只有玩家附近的 chunk 需要存在；远处的 chunk 没加载、不存在。绝大多数 chunk 槽位是空的。

用 3D 数组意味着要为"整个世界的所有 chunk 位置"预分配内存，其中 99% 是空槽，纯粹浪费。而 Dictionary<long, MCChunk> 只存实际存在的 chunk：玩家走到哪，就把附近的 chunk 塞进字典；走远了，就把远处的 chunk 从字典移除、数组回池。字典的 key 是 chunk 的唯一编号（long），value 是 chunk 实例。这样：

3D 数组方案（弃用）：
  ┌─────────────────────────────────────┐
  │ 固定大小的立方体网格，绝大多数槽位空 │  ← 必须为整个世界预分配，不可行
  │   玩家附近少数有 chunk，其余全是 null │
  └─────────────────────────────────────┘

Dictionary<long, MCChunk> 方案（采用）：
  Chunks = { chunkIndex₁ → chunk₁,
             chunkIndex₂ → chunk₂,
             chunkIndex₃ → chunk₃, ... }    ← 只存实际加载的 chunk
  玩家移动 → 加新 chunk 进字典、移除远的 chunk  ← 按需加载/释放，不受总尺寸限制

代价是查询时要算一次 chunkIndex 再查字典（O(1) 哈希查找），而不是数组那种直接下标寻址。但相比"开不出内存"和"绝大多数空槽"，这点哈希开销完全可接受。这就是"用一维字典存 3D 网格"的设计意图：稀疏 + 动态加载，用字典按需存取，不受世界总尺寸限制。

4.2 chunkIndex 这个 long 怎么编码（重点）

字典的 key 是 long，它把 chunk 的三维整数索引 (x, y, z) 压成一个 64 位整数。编码在 NetWorkUtility.ConvertXYZ_20（Packages/com.unity.mccore/Runtime/Utils/NetWorkUtility.cs:202）：

public static long ConvertXYZ_20(int x, int y, int z)
{
    long convertNum = (CompressWithServerNum_20(x)) << 40
                    | (CompressWithServerNum_20(y)) << 20
                    | (CompressWithServerNum_20(z));
    return convertNum;
}

// 单个分量压缩：20 位，1-19 位是数值绝对值，第 20 位（0x80000）是符号位
public static long CompressWithServerNum_20(long pNumber)
{
    bool isMinus = pNumber < 0;
    if (isMinus)
    {
        pNumber = Math.Abs(pNumber);
        pNumber = pNumber | 0x80000;   // 第 20 位置 1 表示负
    }
    return pNumber;
}

把 64 位 long 拆成三段，每段 20 位，分别装 x、y、z：

  bit:  63                    40 39                    20 19                     0
       ┌─────────────────────────┬─────────────────────────┬─────────────────────────┐
       │      x（20 位）         │      y（20 位）         │      z（20 位）         │
       └─────────────────────────┴─────────────────────────┴─────────────────────────┘

  每段 20 位内部：
    bit 19        bit 18 ... bit 0
    ┌──────┬─────────────────────┐
    │ 符号 │  数值绝对值 (19 位)  │
    │ 0正  │  0 ~ 524287         │
    │ 1负  │                     │
    └──────┴─────────────────────┘

  → 单个方向支持 -524287 ~ +524287，三段合起来正好占 60 位，落在 long(64位) 里。

所以 chunkIndex = (compressX << 40) | (compressY << 20) | compressZ。三个分量各自带符号位，解码时按段拆开再还原符号即可。这套编码是客户端与服务器交互约定（文件注释原话），网络包里传 chunk 坐标用的也是它，字典 key 复用同一套编码省得再转一次。

MCWorld 里所有需要 chunkIndex 的地方都走这条链：要么直接调 NetWorkUtility.ConvertXYZ_20(x,y,z)，要么调 YKVector3i 的扩展方法 Convert_20()（NetWorkUtility.cs:62，内部就是转发到 ConvertXYZ_20）。例如 TryGetChunk(YKVector3i index)（MCWorld.cs:210）就是 long index_17 = index.Convert_20(); return Chunks.TryGetValue(index_17, out chunk);。

核实发现的注释不一致：方法名后缀 _20 表示"每段 20 位"，代码也确实是 20 位（符号位 0x80000 = 1<<19，占第 20 位）。但同文件里 CompressWithServerNum_20 上方的注释写的是"xyz 分别用 17 位表达"（NetWorkUtility.cs:143），这是过时注释——历史上可能用过 17 位（MCWorld.cs 里还有一组 CompressWithNum_17 / ToVector3_17 的遗留辅助函数，MCWorld.cs:266 起，目前 chunk 字典 key 没在用）。以实际代码为准：当前 chunkIndex 编码是 20 位/段，共 60 位。

4.3 世界坐标 → chunkIndex 的换算

有了编码方式，再看 MCWorld 怎么从一个世界坐标算出 chunkIndex。核心方法是 PositionToIndex(float, float, float, out long chunkIndex, out int blockIndex)（MCWorld.cs:182）：

public void PositionToIndex(float worldX, float worldY, float worldZ,
                            out long chunkIndex, out int blockIndex)
{
    // 1. 世界坐标 → chunk 整数索引（三个分量）
    int x = (int)Math.Floor(worldX * ChunkSizeMulBlockSizeInverse);
    int y = (int)Math.Floor(worldY * ChunkSizeMulBlockSizeInverse);
    int z = (int)Math.Floor(worldZ * ChunkSizeMulBlockSizeInverse);

    // 2. 三个 int 压成 long，作为字典 key
    chunkIndex = NetWorkUtility.ConvertXYZ_20(x, y, z);

    // 3. 查字典拿到 chunk，再让 chunk 算 block 在数组里的下标
    blockIndex = -1;
    if (Chunks.TryGetValue(chunkIndex, out var chunk))
    {
        blockIndex = chunk.PositionToArrayIndex(worldX, worldY, worldZ);
    }
}

这里的 ChunkSizeMulBlockSizeInverse = 1 / (ChunkSize × BlockSize)，也就是"一个 chunk 世界边长的倒数"。世界坐标乘它再向下取整，直接得到 chunk 的整数索引——因为一个 chunk 边长就是 ChunkSize × BlockSize，世界坐标除以这个边长就是"第几个 chunk"。

注意这一步除的是 chunk 边长（32×BlockSize），不是 BlockSize。这一点容易混：MCWorld.PositionToIndex 算的是 chunk 索引（除以 chunk 边长），MCChunk.PositionToArrayIndex 算的是 chunk 内 block 索引（除以 BlockSize）。两个尺度不同，分工明确：

世界坐标 (worldX, worldY, worldZ)
   │
   │  ÷ (ChunkSize × BlockSize) = ÷ chunk 边长   ← MCWorld 这一层做
   ▼
chunk 整数索引 (x, y, z)   →  ConvertXYZ_20  →  chunkIndex (long)  →  字典查 chunk
   │
   │  chunk 内部：(worldPos - Offset) ÷ BlockSize   ← MCChunk 这一层做
   ▼
block 数组下标 (0 ~ 32767)  →  chunk[blockIndex]  →  MCBlock

如果 chunk 不在字典里（没加载），blockIndex 返回 -1，调用方就知道"这个坐标当前没数据"。

方法行号作用

5. 从世界坐标到方块的完整查询链

前三节分别讲了 MCBlock、MCChunk、MCWorld。这一节把它们串成一条完整的查询链：给一个世界坐标，怎么一步步拿到那个方块。这是本篇的收束点。

5.1 完整流程图

调用方 (WorldSceneSystem)
   │
   │  给一个世界坐标 (worldX, worldY, worldZ)
   ▼
┌──────────────────────────────────────────────────────────────────┐
│ MCScene.MCManager.CurrentMCWorld                                 │
│   （取当前 LOD0 的 MCWorld，即 m_Worlds[0]）                      │
└──────────────────────────────────────────────────────────────────┘
   │
   │  PositionToIndex(worldX, worldY, worldZ, out chunkIndex, out blockIndex)
   ▼
┌──────────────────────────────────────────────────────────────────┐
│ ① worldPos ÷ (ChunkSize×BlockSize) 取整 → chunk 索引 (x,y,z)     │
│ ② ConvertXYZ_20(x,y,z) → chunkIndex (long)                       │
│ ③ Chunks.TryGetValue(chunkIndex, out chunk)                      │
│    ├─ 没找到 → block = MCBlock.Empty，返回 false                  │
│    └─ 找到  → chunk.PositionToArrayIndex(worldPos) → blockIndex   │
└──────────────────────────────────────────────────────────────────┘
   │
   ▼
┌──────────────────────────────────────────────────────────────────┐
│ chunk.Blocks[blockIndex]  →  MCBlock (struct，值拷贝出来)        │
│    Type / BlockMask / CuttingType                                │
└──────────────────────────────────────────────────────────────────┘
   │

   ▼

三步走：选层（CurrentMCWorld） → 算 chunkIndex + 查字典拿 chunk → 算 blockIndex + 取数组元素。每一步只解决一个问题，层层递进。

5.2 WorldSceneSystem.TryGetBlockInternal —— 不带缓存的版本

业务侧最基础的查询入口在 WorldSceneSystem.TryGetBlockInternal（Assets/Scripts/Game/GameWorld/Game/OpenWorldGameSystem/WorldSceneSystem.cs:605）：

private bool TryGetBlockInternal(float x, float y, float z, out MCBlock block)
{
    if (MCScene == null || MCScene.MCManager == null || MCScene.MCManager.CurrentMCWorld == null)
    {
        block = MCBlock.Empty;
        return false;
    }
    if (MCScene.MCManager.CurrentMCWorld.TryGetBlock(x, y, z, out block))
    {
        return !MCObjectUtils.IsEmptyBlock(block.Type);   // 字典里拿到 chunk，但 block 可能是空块
    }
    block = MCBlock.Empty;
    return false;
}

它直接调 MCWorld.TryGetBlock，把"算 chunkIndex → 查字典 → 算 blockIndex → 取 block"全包在 TryGetBlock 里。最后用 IsEmptyBlock 再判一次，是因为 chunk 里也可能存着 Type==0 的空块——字典命中不等于有实际方块。

5.3 TryGetBlockUseCache —— 带缓存的版本（重点）

体素查询有个特点：调用方往往连续查同一片区域的方块（比如构建一个 chunk 的 mesh 时要遍历它周围所有方块、或者射线检测沿一条线逐格查）。这些查询大概率落在同一个 chunk 内，每次都走一遍"算 chunkIndex → 字典查找"有点浪费，因为 chunkIndex 算出来一样、字典查出来的还是同一个 chunk。

于是 WorldSceneSystem.TryGetBlockUseCache（WorldSceneSystem.cs:634）加了一层单 chunk 缓存：

// 三个字段都是 WorldSceneSystem 的成员
private long m_ChunkIndexCache;       // 上次查的 chunkIndex
private MCChunk m_McChunkCache;       // 上次查到的 chunk

public bool TryGetBlockUseCache(ref YKVector3d ckPos, ref MCBlock block)
{
    if (MCScene == null || MCScene.MCManager == null || MCScene.MCManager.CurrentMCWorld == null)
    {
        block = MCBlock.Empty;
        return false;
    }
    // 每次都要算 chunkIndex（这步避不开）
    MCScene.MCManager.CurrentMCWorld.PositionToIndex(ckPos, out long chunkIndex, out int blockIndex);

    if (m_McChunkCache != null && chunkIndex == m_ChunkIndexCache)
    {
        // 命中缓存：同一个 chunk，直接用缓存的 chunk 取 block，跳过字典查找
        block = ref m_McChunkCache[blockIndex];
        return !MCObjectUtils.IsEmptyBlock(block.Type);
    }
    else
    {
        // 未命中：查字典拿 chunk，并更新缓存
        if (MCScene.MCManager.CurrentMCWorld.TryGetChunk(chunkIndex, out m_McChunkCache))
        {
            m_ChunkIndexCache = chunkIndex;
            block = ref m_McChunkCache[blockIndex];
            return !MCObjectUtils.IsEmptyBlock(block.Type);
        }
    }
    block = MCBlock.Empty;
    return false;
}

缓存逻辑很简单：

  1. 每次都算 chunkIndex 和 blockIndex（这步避不开，因为要靠 chunkIndex 判断是否命中）。
  2. 如果上次缓存的 chunk 不为空 且 这次算出的 chunkIndex 和上次一样 → 命中，直接用 m_McChunkCache 取 block，跳过 TryGetChunk 的字典查找。
  3. 否则未命中，走 TryGetChunk 查字典，查到后把 chunk 和 chunkIndex 都缓存下来，给下次用。

为什么这个缓存有效？因为体素查询天然有局部性：遍历一个 chunk 的 32768 个方块时，它们的 chunkIndex 全相同，只有 blockIndex 变，缓存命中率几乎 100%；跨 chunk 边界时才 miss 一次重新加载。对"按 chunk 批量处理方块"这种高频场景，省下的字典查找开销很可观。

有两个约束要记住：

  • 单线程使用：代码注释三处都写明"带缓存的 block 查询器，单线程使用"（WorldSceneSystem.cs:622-630）。m_McChunkCache 和 m_ChunkIndexCache 是普通字段，没有任何同步保护，多线程并发用会互相覆盖缓存。实际调用都在主线程或确定单线程的路径上。
  • 只缓存"最近一个 chunk"：这是个单槽缓存，不是 LRU。一旦跨到别的 chunk，旧缓存立刻被覆盖。所以它对"来回跳着查两个 chunk"的场景没有加速效果——但体素查询通常不会这么用。
为什么用 ref 返回 block：注意 block = ref m_McChunkCache[blockIndex] 用了 ref。因为 MCBlock 是 struct，不用 ref 的话这里会做一次值拷贝；用 ref 返回的是数组里那个方块的引用，调用方既能读也能直接改。配合 ref 参数签名，整个查询是零拷贝的。

5.4 两个版本怎么选

方法缓存适用场景
TryGetBlockInternal无偶尔查一次、查完就走的场景（比如 Lua 侧问"这个坐标有方块吗"）
TryGetBlockUseCache单 chunk批量、连续查同一片区域的场景（mesh 构建、射线检测、邻居查询）

这就是本篇的完整闭环：从"一个世界坐标"到"那个方块的字段"，中间穿过 MCWorld 的字典、MCChunk 的数组、MCBlock 的 struct 布局，每一步都有明确的数据结构和换算关系。

6. 小结与下一站

本篇要点回顾

  1. 三层存储模型：世界 → 多层 MCWorld → 每层一堆 MCChunk → 每个 chunk 是 32³ 个 MCBlock。本篇只讲"单层 MCWorld 内部怎么存"，5 层 LOD 设计第 3 篇展开。全部代码在 Packages/com.unity.mccore/Runtime/MCWorld.cs。
  2. MCBlock 是 struct（6 字节，LayoutKind.Explicit + Pack=1）：Type(ushort) + BlockMask(ushort) + CuttingType(byte)。用 struct 是为了让海量方块内存连续、紧凑、零 GC、缓存友好。Type==0 是空块，MCBlock.Empty 是查询失败的统一返回值。
  3. MCChunk 持有 MCBlock[32768]，坐标→下标换算用 (x*32 + y)*32 + z（行主序，z 最内层连续）。脏标记用一个 byte 的位标志，分"方块脏"和"实体脏"两条通道。MCBlock[] 走 MCBlockPool 对象池复用，避免反复分配 32768 长度的数组。
  4. MCWorld 用 Dictionary<long, MCChunk>，因为大世界稀疏且动态加载，3D 数组开不出内存。chunkIndex 把三维索引压成 long，每段 20 位（符号位+19 位数值），共 60 位，编码方式是客户端/服务器约定。
  5. 完整查询链：CurrentMCWorld → PositionToIndex 算 chunkIndex+blockIndex → 字典查 chunk → chunk[blockIndex] 取 block。TryGetBlockUseCache 加了一层单 chunk 缓存，利用体素查询的局部性跳过字典查找，但只能单线程用。

承上启下

这一篇把"方块怎么存"讲透了：MCBlock 是最小单元、MCChunk 是 32³ 的方块组、MCWorld 是稀疏字典容器，三者用 struct 数组 + 一维字典 + 坐标编码串起来。但这只是静态的存储结构——一个 chunk 什么时候被创建出来塞进字典、什么时候被移除、5 层 MCWorld 怎么协作、玩家移动时谁来调度这些，都还没讲。
