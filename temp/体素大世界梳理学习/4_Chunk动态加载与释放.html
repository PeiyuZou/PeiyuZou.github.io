1. 动态加载整体流程总览

第 1 篇讲过 MCScene.SetLocation 是驱动重加载的总开关：玩家位置变化超过一个 chunk（32 米）时，切换到 loading 态，在工作线程上重新扫描。本篇从这之后接着讲——SetLocation 触发重加载后，引擎内部怎么一步步把"该加载哪些 chunk"算出来、把数据填进去、把 mesh 建好、把不用的 chunk 卸掉。

这条链上有几个关键设计点，是本篇要讲透的：

  • 分片调度：世界不是一整块，而是切成一个个 Patch（TD 块，512×512 米），每个 Patch 自己管一片区域的 chunk 加载。这样可以按区域分别调度、控制单帧加载量。
  • 差集比对：每帧重扫描时，不是"推倒重来"，而是把"该存在的 chunk 集合"和"当前已加载的 chunk 集合"做差集，只处理增量。
  • 脏标记驱动：chunk 不是每次都全量重建 mesh，而是靠脏标记只重建变动的那部分。
  • mesh 分帧构建：一个 chunk 32768 个 block，一次建完会卡，所以 mesh 构建扔到工作线程按 LOD 分队列、限流消费。

2. Patch 与 MCPatch —— chunk 加载的调度单元

先回答一个自然会问的问题：为什么不直接用一个全局列表管所有 chunk，而要先切成 Patch？

体素大世界很大（几公里见方），chunk 数量动辄上万。如果只有一个全局列表，玩家每移动一次就要对全量 chunk 做一次"该不该加载"的判定，单帧成本会随世界尺寸线性增长。更麻烦的是分帧分摊——你想"这一帧只加载 3 个 chunk"，但全局列表里 chunk 混在一起，你很难按"区域远近"来排队。所以引擎把世界切成一个个 Patch（也叫 TD 块，每块 512×512 米），让调度按 Patch 分片进行。

2.1 MCPatch 是什么

MCPatch（Assets/Scripts/Core/MCCore/Scene/MCPatch.cs:208）继承自引擎层的 TiledPatch，是体素侧的 Patch 实现。一个 MCPatch 管一片 512×512 米区域的 chunk 加载，核心持有三样东西：

public class MCPatch : TiledPatch
{
    protected Octree m_Octree;            // 这片区域的八叉树（覆盖 8192 见方，最小划分到 32）
    protected OctreeQueryByLod m_Query;   // 八叉树查询器（持有 _pos 观察点 + _lodDis[] 各层距离）
    // ...
    public const int ChunkSize = 32;
}

注意 m_Octree 不是只装 512×512 这一小块的 chunk——它在 InitOctree（MCPatch.cs:795）里建的是一棵覆盖 8192 见方的八叉树（size = 8192，maxdepth 一路除到 32 为止）。也就是说每个 Patch 的八叉树其实比 Patch 自身大得多，这是为了容纳跨 Patch 边界的 chunk 数据。每个 chunk 在八叉树里是一个节点（ChunkEntity），按它的中心坐标和 size（blockSize×32）落到对应深度的位置。

2.2 MCManager.PatchesCache —— Patch 的容器

MCManager（Assets/Scripts/Core/MCCore/Scene/MCManager.cs:148）持有 PatchesCache，里面是一个 ConcurrentDictionary<long, TiledPatch>，key 是 Patch 的 UV 编码（TiledScheduler.GetBlockIndex(u, v)，把 u/v 压进一个 long）。所有当前已加载的 Patch 都在这里。

// ISchedulable.cs:28
public class PatchesCache
{
    public ConcurrentDictionary<long, TiledPatch> Patches { get; }
}

// MCManager.cs:136
PatchesCache = new PatchesCache();

所以分片调度的层次是：MCManager → PatchesCache → 多个 MCPatch → 每个 MCPatch 一棵八叉树 → 八叉树里挂 ChunkEntity。玩家移动时，先按距离决定哪些 Patch 该进/该出（粗筛），再在每个 Patch 内部按 LOD 距离决定哪些 chunk 该加载（细筛）。

2.3 OctreeQueryByLod —— 八叉树空间查询器

OctreeQueryByLod（Assets/Scripts/Core/MCCore/Octree/Octree.cs:650）是"按 LOD 距离做范围查询"的查询器，结构极简：

public class OctreeQueryByLod
{
    public YKVector3d _pos;        // 观察位置（当前中心点）
    public float[] _lodDis;        // 各八叉树深度对应的查询距离
}

核心方法 Contain(OctreeNode node)（Octree.cs:675）判定一个节点该不该被纳入扫描结果：取节点所在深度对应的距离阈值 _lodDis[node.Depth]，若节点中心到观察点三轴距离都 ≤ 这个阈值，就纳入。

public bool Contain(OctreeNode node)
{
    float fRange = _lodDis[node.Depth];   // 按深度取距离
    float disX = Math.Abs((float)(node.Center.x - _pos.x));
    float disY = Math.Abs((float)(node.Center.y - _pos.y));
    float disZ = Math.Abs((float)(node.Center.z - _pos.z));
    return (disX <= fRange && disY <= fRange && disZ <= fRange);
}
呼应第 3 篇：_lodDis 数组下标是八叉树深度，不是 LOD 层号。第 3 篇讲过八叉树从根（depth 0，size 8192）一路分到叶子（depth 8，size 32），叶子节点正好对应 LOD0 的 chunk（blockSize=1，size=32）。所以 _lodDis[8] 是 LOD0 的查询距离（近，比如 96 米），_lodDis[4] 是 LOD4 的查询距离（远，比如 2048 米）。深度越浅→节点越大→距离阈值越大，自然形成"近处用小 chunk、远处用大 chunk"的同心圆。SetQueryLodDistance（MCPatch.cs:857）按画质档位（LowMemory/Low/Medium/High/Ultra）填充这 5 个槽位。

八叉树的 Query（Octree.cs:335）就是递归调用 Contain：从根开始，当前节点在范围内才递归子节点；如果某个节点的子节点有被纳入的，就把同层兄弟节点也都补齐（保证八叉树某一层要么全选要么不选，避免出现"一个父节点只选了一半孩子"的破洞），最终把命中的叶子层 ChunkEntity 收进结果列表。

3. 加载判定：哪些 chunk 该存在

这是本篇最核心的设计。重加载不是"把当前的 chunk 全卸了再重装"，而是做一次差集比对：用当前 Location 重新查询"应该存在的 chunk 集合"，和"当前已加载的 chunk 集合"做对比，只处理两边不一样的部分。

3.1 差集比对的设计

逻辑在 MCPatch.UpdateLod（MCPatch.cs:408）。它用的是双缓冲 + 标志位的技巧，没有显式建两个 HashSet 再求差集，而是用一个 LoadFlag 标志位配合两个 list 来完成：

本轮扫描前：
  listBuffer1 = []              ← 准备装本轮扫描结果
  listBuffer2 = [上一轮的 chunk] ← 每个 chunk 都标着 LoadFlag=true

Step1: m_Octree.Query(m_Query, listBuffer1)
  listBuffer1 = [本轮该存在的 chunk]  (含上一轮残留的 ChunkEntity + 新建的)

Step2: 遍历 listBuffer1，对每个 chunk:
  source.SetFlag(LoadFlag, true)   ← 标记"本轮我还在"

  如果之前没标 IsRender（说明是新进的）:
    → 加入 m_AddEntities (待加载)
    → SetFlag(IsRender, true)

  如果之前已 IsRender（说明是老朋友）:
    → 检查是否 NeedRefresh，需要则触发 OnChunkRefresh

Step3: 遍历 listBuffer2（上一轮的 chunk）:
  对每个 LoadFlag 仍是 false 的（说明本轮没被扫到）:
    → SetFlag(IsRender, false)
    → 加入 m_RemoveEntities (待卸载)

Step4: 交换 listBuffer1 ↔ listBuffer2
  清掉 listBuffer2（现在的旧轮）里 chunk 的 LoadFlag
  下一轮 listBuffer1 又是空的，listBuffer2 装着本轮结果

关键代码片段：

// MCPatch.cs:426 — 设置观察点并查询
m_Query._pos = ((MCScene)Scene).IsSkiBattle ? scene.Location : PosToChunkCenter(scene.Location);
m_Octree.Query(m_Query, listBuffer1);

// MCPatch.cs:454 — 本轮扫到的，先标 LoadFlag
source.SetFlag((byte)ChunkEntityFlag.LoadFlag, true);

// MCPatch.cs:474 — 之前没渲染的 → 新增
if (!source.CheckFlag((byte)ChunkEntityFlag.IsRender))
{
    source.SetFlag((byte)ChunkEntityFlag.IsRender, true);
    source.LoadingStates = LoadingSessionStates.Loading;
    source.SetFlag((byte)ChunkEntityFlag.ScanState_Add, true);
    m_AddEntities.Add(source);
}

// MCPatch.cs:569 — 旧轮里 LoadFlag 被清的 → 卸载
if (!source.CheckFlag((byte)ChunkEntityFlag.LoadFlag))
{
    if (source.CheckFlag((byte)ChunkEntityFlag.IsRender))
    {
        source.SetFlag((byte)ChunkEntityFlag.IsRender, false);
        source.SetFlag((byte)ChunkEntityFlag.ScanState_Remove, true);
        source.LoadingStates = LoadingSessionStates.Unloading;
        m_RemoveEntities.Add(source);
    }
}

// MCPatch.cs:654 — 双缓冲交换
(listBuffer1, listBuffer2) = (listBuffer2, listBuffer1);
for (int i = 0; i < listBuffer2.Count; i++)
{
    ChunkEntity source = listBuffer2[i] as ChunkEntity;
    if (source != null) source.SetFlag((byte)ChunkEntityFlag.LoadFlag, false);
}
设计意图：这个"双缓冲 + LoadFlag"的玩法，本质上就是把新集合 ∪ 旧集合 → 加载/卸载两边这个差集运算，摊到了两次遍历里，避免了建 HashSet 做 Except 的开销。ChunkEntity 是常驻对象（项目里常有 17 万个），给它加一个 byte 标志位比维护两个集合便宜得多。注意它不做 List.Remove（MCPatch.cs:459 注释明说"避免消耗"），不可见的 chunk 直接把 list 槽位置 null，下次交换后自然丢掉。

3.2 距离阈值：5 层 LOD 的同心圆

"该存在"的判定标准就是 OctreeQueryByLod.Contain 里的距离比较。各 LOD 有自己的查询距离（_lodDis），近层距离小、远层距离大，形成同心圆：

                _lodDis[4] = 2048  ── LOD4 (blockSize 16, 最远, 最粗)
           ┌─────────────────────────────────┐
           │      _lodDis[5] = 992  ── LOD3   │
           │   ┌───────────────────────────┐  │
           │   │    _lodDis[6] = 480 ── LOD2│  │
           │   │  ┌─────────────────────┐   │  │
           │   │  │  _lodDis[7]=224 LOD1│   │  │
           │   │  │ ┌───────────────┐   │   │  │
           │   │  │ │ _lodDis[8]=96 │   │   │  │
           │   │  │ │   LOD0 精细    │   │   │  │
           │   │  │ │   ★玩家       │   │   │  │
           │   │  │ └───────────────┘   │   │  │
           │   │  └─────────────────────┘   │  │
           │   └───────────────────────────┘  │
           └─────────────────────────────────┘
        (Medium 画质档的数值，来自 MCPatch.cs:888)

玩家走动时，这个同心圆跟着玩家移动。某个 LOD0 chunk 走出 96 米圈 → 不在 _lodDis[8] 范围内 → Contain 返回 false → 本轮没被扫到 → LoadFlag 被清 → 进 m_RemoveEntities。反之远处一个 LOD4 chunk 走进 2048 米圈 → 被扫到 → 进 m_AddEntities。这就是"哪些 chunk 该存在"的判定全貌。

网络数据填充只点一下：chunk 的方块数据分两路：本地 TD 文件解压 + 服务器推送合并。本篇讲的是客户端"判定要加载哪些 chunk"这条调度链，网络数据怎么来、怎么合到 chunk 里是第 8/10 篇的内容。这里只要知道：chunk 被判定要加载后，OnChunkAdded 里会触发 LoadChunkData → Uncompress 把数据填进 MCBlock[]，服务器推送的变更 chunk 通过 ServerChangedChunks 字典登记、ProcessNetData 入树（MCPatch.cs:1867）。

4. chunk 的加载与数据填充

一个 chunk 被判定要加载（进了 m_AddEntities）后，UpdateLod 末尾会把它派发给主线程执行 OnChunkAdded（MCPatch.cs:994 → Manager.OnChunkAdded）。加载流程在 MCManager.OnChunkAdded（MCManager.cs:757）：

// MCManager.cs:757
public virtual void OnChunkAdded(ChunkEntity source, bool isAuxiliary = false)
{
    source.SetFlag((byte)ChunkEntityFlag.Visible, true);

    if (source.Chunk == null)
    {
        // Chunk 还没有数据 → 加载
        source.LoadChunkData(() => {
            Scene.ExecuteInThread(new UniversalScene.ThreadTask(source, (obj) => {
                (obj as ChunkEntity)?.Uncompress();              // 工作线程解压
                OnChunkUncompressed(obj as ChunkEntity, ...);    // 回主线程 AddChunk
            }));
        });
    }
    else
    {
        AddChunk(source, source.Chunk);  // 已有数据，直接渲染
    }
}

4.1 MCBlock[] 从哪来

第 2 篇讲过 MCChunk 持有 MCBlock[] Blocks，长度恒为 32×32×32 = 32768。这个数组从对象池来：MCChunk 构造函数（Packages/com.unity.mccore/Runtime/MCWorld.cs:652）调 MCBlockPool.AllocateMCBlocks() 取一个池化数组，而不是 new 一个新的。

// MCWorld.cs:652
public MCChunk(float blockSize, int chunkSize, YKVector3f offset)
{
    Blocks = MCBlockPool.AllocateMCBlocks();   // 从池取
    // ...
    IsDirty = MCChunkDirtyFlags.ChunkAllDirty; // 首次创建置全脏
}

// MCWorld.cs:682
public void Dispose()
{
    if (!_isDisposed) {
        MCBlockPool.RecycleMCBlocks(Blocks);   // 回池
        _isDisposed = true;
    }
}

MCBlockPool（MCWorld.cs:458）是个静态 Queue<MCBlock[]>，默认 512 个数组，低画质 ×0.5（MCManager.cs:128）。AllocateMCBlocks 池空了才 new，RecycleMCBlocks 回收时 Array.Clear 清零再入队。这样 chunk 频繁进出时不会反复分配 32768 长度的数组，GC 压力大幅降低。

4.2 数据填充与登记

解压后的 MCChunk 通过 OnChunkUncompressed → AddChunk（MCManager.cs:994）登记到 MCWorld：

  • 登记到 MCWorld.Chunks：LOD0 走 SetMCWorldChunk（MCManager.cs:976），把 chunk 按它的 index（ConvertXYZ_20 编码成 long）塞进 m_Worlds[0].Chunks 字典。若已有旧 chunk，先 Dispose 旧的再覆盖。
  • 置脏标记：MCChunk 构造时 IsDirty = ChunkAllDirty（全脏），意味着这个 chunk 的 mesh 第一次必须完整构建。
  • 触发渲染：chunkEntity.Render(chunk, this) 最终走到 RendererChunk（MCManager.cs:1084），MarkAllDirty() 后构造 ChunkRenderWrap 丢进图形队列。
// MCManager.cs:994
public virtual void AddChunk(ChunkEntity chunkEntity, MCChunk chunk)
{
    int index = (int)Mathf.Log(chunkEntity.BlockSize, 2);
    if (index == 0) SetMCWorldChunk(chunk);   // LOD0 登记进 MCWorld.Chunks
    chunkEntity.Render(chunk, this);           // 触发 mesh 构建
}

// MCManager.cs:976
public void SetMCWorldChunk(MCChunk chunk)
{
    long indexLong = NetWorkUtility.ConvertXYZ_20(chunk.Index.x, chunk.Index.y, chunk.Index.z);
    if (m_Worlds[0].Chunks.TryGetValue(indexLong, out var mcChunk))
    {
        if (mcChunk == chunk) return;
        mcChunk?.Dispose();                    // 旧 chunk 回池
        MCBlockCollision.SetDirty(indexLong);
    }
    m_Worlds[0].Chunks[indexLong] = chunk;     // 登记新 chunk
}
为什么 LOD0 才登记进 MCWorld.Chunks：第 3 篇讲过 5 层 MCWorld 各自存各自的 chunk。但业务查询（TryGetBlock、SetBlock）默认走 LOD0，所以加载时 LOD0 的 chunk 必须登记进 m_Worlds[0].Chunks 这个字典，业务层才能查到。高层 LOD 的 chunk 主要用于渲染远处的粗略地形，不参与方块级查询，登记与否取决于具体路径（RendererChunk 里直接走渲染队列，不一定塞 Chunks 字典）。

5. mesh 构建与分帧调度

chunk 登记完就要建 mesh。但mesh 构建不能在主线程一次性做完——一个 chunk 32768 个 block，逐个判定面剔除、合并网格，单 chunk 建完就要几毫秒，玩家移动一下可能要建几十个 chunk，全堆主线程会卡死。所以 mesh 构建被拆到独立工作线程 + 按 LOD 分队列 + 限流消费。

5.1 图形队列 MCGraphicsQueue

MCGraphicsQueue（Assets/Scripts/Engine/Graphics/MCGraphicsQueue.cs:17）是 mesh 构建的调度器。它内部有5 个按 LOD 分的并发队列，一个独立后台线程：

// MCGraphicsQueue.cs:42

5.2 限流消费：为什么分帧

后台线程的 GetRenderTask（MCGraphicsQueue.cs:144）每轮只取固定数量的任务出来执行：

// MCGraphicsQueue.cs:144
private void GetRenderTask(ConcurrentQueue<RenderThreadTask> result)
{
    // 同时受 RENDER_TASK_MAX 和 mesh 数据缓冲池剩余量双重限制
    int doRenderCount = Math.Min(RENDER_TASK_MAX, MCBuffersMeshData.MeshRenderDataPoolCount());
    if (doRenderCount <= 0) return;

    for (int i = 0; i < m_ThreadExecuteTask.Length; i++)
    {
        var renderTaskQueue = m_ThreadExecuteTask[i];
        while (renderTaskQueue.TryDequeue(out var task))
        {
            result.Enqueue(task);
            if (result.Count >= doRenderCount) return;   // 取够就停
        }
    }
}
双重限流的意图：一是 RENDER_TASK_MAX 限制单轮构建数量，避免后台线程一口气把堆积的几十个 chunk 全建完（建完还要往主线程推 GameObject，主线程挂对象也是开销）；二是 MeshRenderDataPoolCount() 限制——如果主线程还没消费掉已建好的 mesh 数据（缓冲池空了），后台就停下不再建，防止内存炸裂（注释原话 MCGraphicsQueue.cs:146）。这两个限流共同把 mesh 构建摊到多帧，玩家看到的就是"远处 chunk 一片片逐渐出现"而不是一次性卡顿后全出现。

真正建网格在 MCManager.BuildMeshEx（MCManager.cs:1144），由后台线程通过 CreateRenderData 触发：

6. chunk 的卸载与资源释放

chunk 移出加载范围后（进 m_RemoveEntities），UpdateLod 末尾派发 OnChunkRemoved（MCPatch.cs:1002 → Manager.OnChunkRemoved，MCManager.cs:1384）。卸载比加载更强调彻底——chunk 走了，它的 mesh、方块数据、游戏对象、服务器 entity 引用都得干净地回收，否则玩家来回跑会内存泄漏。

6.1 卸载的几条清理路径

OnChunkRemoved (MCManager.cs:1384)
   │
   ├─ source.OnChunkRemoved(m_Worlds[index])
   │     └─ chunkEntity.ReleaseMCChunkImplement(world)

   │                                              MCChunk.Dispose() → MCBlock[] 回池

   ├─ m_OnChunkRemoveAction?.Invoke(source)    ── 外部注册的清理回调
   │     (WorldSceneSystem 注册的服务器 entity 清理走这条)
   │
   └─ RemoveGameObject(chunk) (MCManager.cs:1519)
         ├─ RemoveChunkEntityObject (仅 LOD0)   ── 异形块/装饰 entity 回收
         │     └─ entity.OnRelease + scene.RemoveEntity
         └─ RemoveChunkBlockObject              ── chunk 的 mesh GameObject 回收
               └─ item.Clear() + StoreMCChunkGameObject (对象池)

6.2 MCBlock[] 回池

6.3 服务器 entity 联动清理

第 10 篇会展开网络同步，这里点出联动：WorldSceneSystem.LoadScene（Assets/Scripts/Game/GameWorld/Game/OpenWorldGameSystem/WorldSceneSystem.cs:185）在场景加载时注册：

意思是：chunk 被引擎卸载时，按 chunk 维护的服务器 entity 数据（树、石头这些物件的合并数据）也要一起清掉。这就是 m_OnChunkRemoveAction 这条外部回调链的用途——让业务层能挂在 chunk 卸载事件上做联动清理，而不需要引擎层去懂业务。

卸载彻底性的要点：玩家来回跑同一片区域时，chunk 反复加载/卸载。如果卸载时漏了任何一环——比如忘了从 m_Worlds[0].Chunks 移除、忘了把 MCBlock[] 回池、忘了清服务器 entity 引用——每次往返都会留下一份悬挂数据，内存就一点点涨上去。所以卸载路径是多条清理同时进行（字典移除 + 数组回池 + GameObject 回收 + 外部回调），任何一条都不能少。代码里 MCPatch.OnRelease（MCPatch.cs:1160）和 OnDestroy（MCPatch.cs:2065）还做了整 Patch 级别的兜底清理（RemoveAllChunk + RemoveAllEntity），保证 Patch 整体卸载时不会留尾巴。

7. 脏标记驱动的局部刷新

第 2 篇讲了 MCChunk 有个 IsDirty byte 位标志，定义在 MCChunkDirtyFlags（Packages/com.unity.mccore/Runtime/MCWorld.cs:557）：

public static class MCChunkDirtyFlags
{
    public const byte ChunkBlockDirty  = 1 << 0;                 // 0x01 方块数据脏
    public const byte ChunkEntityDirty = 1 << 1;                 // 0x02 实体数据脏
    public const byte ChunkAllDirty    = ChunkBlockDirty | ChunkEntityDirty; // 0x03 全脏
}

public class MCChunk
{
    public byte IsDirty = 0;
    public bool IsBlockDirty  => (IsDirty & ChunkBlockDirty) != 0;
    public bool IsEntityDirty => (IsDirty & ChunkEntityDirty) != 0;
    public void MarkBlockDirty()  { IsDirty |= ChunkBlockDirty; }
    public void MarkEntityDirty() { IsDirty |= ChunkEntityDirty; }
    public void MarkAllDirty()    { IsDirty |= ChunkAllDirty; }
    public void ClearAllDirty()   { IsDirty = 0; }
}

本篇讲"谁置脏、谁消费脏"，这是体素世界增量更新的核心机制。

7.1 谁置脏

置脏的本质是"声明这个 chunk 变了，mesh 要重建"。三个主要置脏源：

  • SetBlock 改了某个 block：玩家挖一块、填一块时，MCManager.SetBlock（MCManager.cs:402）改完方块后调 chunk.MarkBlockDirty()。如果新旧方块是异形块（预制体类型），还会 chunk.MarkEntityDirty()（MCManager.cs:453）。
  • chunk 首次创建：构造函数里 IsDirty = ChunkAllDirty（MCWorld.cs:665），首次渲染必须全建。
  • 邻居 chunk 受边界 block 变化连带置脏：见下面 7.3。
// MCManager.cs:452 (SetBlock 里)
chunk.MarkBlockDirty();
if (!chunk.IsEntityDirty)
{
    if (IsBlockPrefeb(chunk.Blocks[blockIndex]) || IsBlockPrefeb(block))
        chunk.MarkEntityDirty();   // 异形块变化要重建实例
}

7.2 谁消费脏

消费脏的是 MCManager.Update（MCManager.cs:1582）。它按帧数模 3 分三件事做，其中 case 2 就是扫脏 chunk：

// MCManager.cs:1582
public override void Update()
{
    m_FrameCount++;
    if (!((ILoadingState)Scene).HasLoadingState(LoadingState.ChunkFinish)) return;

    switch (m_FrameCount % 3)
    {
        case 0: UpdateSetBlock(); break;        // 把暂存的 SetBlock 落到 chunk + 邻居置脏
        case 1: UpdateWaitRemove(); break;
        case 2:                                     // 扫脏 chunk，重建 mesh
            for (int i = 0; i < m_Worlds.Length; i++)
            {
                foreach (var chunk in m_Worlds[i].Chunks.Values)
                {
                    if (chunk.IsDirty != 0)
                        RefreshChunk(chunk, m_Worlds[i]);
                }
            }
            break;
    }
}

// MCManager.cs:676
private void RefreshChunk(MCChunk chunk, MCWorld world)
{
    ChunkEntity source = FindChunkEntity(chunk);
    ChunkRenderWrap wrap = new ChunkRenderWrap(chunk, source, world);
    chunk.ClearAllDirty();                 // 消费脏：清掉标志
    Scene.MCGraphicsT.AddChunkRender(wrap, BuildMeshEx);  // 推进图形队列
}
设计意图：脏标记是"增量更新信号"。没有它，引擎要么每帧全量重建所有 chunk 的 mesh（不可行），要么每次 SetBlock 立刻同步重建那个 chunk 的 mesh（会卡主线程）。有了脏标记，SetBlock 只负责"改数据 + 标脏"（极快），真正的 mesh 重建交给 Update 在 case 2 那帧统一扫脏、推进图形队列、由后台线程分帧建。这样变动被攒成批、摊到多帧、扔到工作线程，主线程几乎无感。注意 case 0/1/2 是每三帧轮一次——即同一个 chunk 标脏后，最多等 3 帧才会被扫到重建，这是用"最长 3 帧延迟"换"主线程零卡顿"的取舍。

7.3 邻居刷新：边界 block 影响 6 邻居

体素 mesh 构建时要判定每个 block 的六个面是不是被邻居挡住了（挡住就不画那个面，这是面剔除的基础）。问题在于：chunk 边界上的 block，它相邻的面可能属于邻居 chunk。如果玩家改了边界上的一个 block，邻居 chunk 那个面的剔除结果也会变，所以邻居 chunk 也得重建 mesh。

改一个边界 block 影响哪些邻居 chunk：

         ┌─────┐
         │ 上  │  (y-1 边界)
         │chunk│
   ┌─────┼─────┼─────┐
   │ 左  │当前 │ 右  │
   │chunk│chunk│chunk│   (x 边界)
   └─────┼─────┼─────┘
         │ 下  │
         │chunk│  (y+1 边界)
         └─────┘
    (z 方向同理，前后各一个)

  排除 8 个角 + 自身 → 最多 18 个方向需要连带置脏
  (角上的 block 面剔除不依赖斜对角邻居，所以 8 角不算)

这个逻辑在 MCManager.CheckRefreshNeighbors（MCManager.cs:857）：

// MCManager.cs:857
public void CheckRefreshNeighbors(MCChunk chunk, int lod, YKVector3d[] dirtyBlockPos)
{
    if (lod != 0 || dirtyBlockPos == null) return;   // 只处理 LOD0

    // 用一个 3×3×3 的 bool 数组标记哪些方向需要标脏
    for (int i = 0; i < dirtyBlockPos.Length; i++)
    {
        YKVector3i dirtyBlockOffset = chunk.PositionToIndex(dirtyBlockPos[i]);
        // 判定这个 block 是不是在 chunk 的 6 个边界上（坐标 0 或 31）
        // 是边界 → 把对应方向的邻居槽位标 true
        // (排除 8 个角，因为角 block 的面剔除不依赖斜对角)
    }

    // 遍历 3×3×3（排除自身和 8 角 = 18 个方向），对标记的邻居 chunk 置脏
    for (int x = -1; x <= 1; x++)
    for (int y = -1; y <= 1; y++)
    for (int z = -1; z <= 1; z++)
    {
        if (Math.Abs(x)==1 && Math.Abs(y)==1 && Math.Abs(z)==1) continue; // 排除 8 角
        if (x==0 && y==0 && z==0) continue;                              // 排除自身
        int flagIndex = ((x+1)*3 + (y+1))*3 + (z+1);
        if (!m_NeighborsChunkRefreshFlags[flagIndex]) continue;

        long chunkIndex = NetWorkUtility.ConvertXYZ_20(chunk.Index.x+x, chunk.Index.y+y, chunk.Index.z+z);
        if (CheckChunkInQueryRange(chunkIndex, lod) && m_Worlds[lod].TryGetChunk(chunkIndex, out var neighborChunk))
            neighborChunk.MarkBlockDirty();   // 邻居置脏
    }
}

UpdateSetBlock（MCManager.cs:474，case 0 那帧执行）就是消费暂存的 SetBlock 列表，对每个被改的 chunk 调 CheckRefreshNeighbors，把受影响的邻居 chunk 一并置脏，下一轮 case 2 扫脏时它们会一起重建。

注意 CheckChunkInQueryRange 这道门：邻居置脏前先判 CheckChunkInQueryRange（MCManager.cs:281），即"这个邻居 chunk 当前是否在加载范围内"。不在范围内的 chunk 根本没加载，置脏也没意义（它下次进加载范围时会因首次创建自动全脏重建）。这道判断避免了给已卸载的 chunk 留悬挂的脏标志。
