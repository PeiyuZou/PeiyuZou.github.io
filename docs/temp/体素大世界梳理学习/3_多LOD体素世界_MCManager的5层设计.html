1. 为什么体素大世界需要多 LOD

第 2 篇我们看清了单层 MCWorld 怎么存方块：一个 chunk 是 32×32×32 = 32768 个 block，世界坐标一路换算到 chunkIndex → blockIndex。但那只解决了"方块怎么存"，没解决一个更扎心的问题：方块太小了，整张地图全用最精细的方块去渲染，根本扛不住。

体素世界里最小方块是米级的（LOD0 的 BlockSize = 1，一个 block 代表 1 米）。一个 32³ 的 chunk 就有 32768 个 block，如果玩家视野能看出去几公里，那远处也按 1 米一个 block 去摆数据、去构网格、去画，数据量和绘制量都会爆炸——你根本不需要在 500 米外看清 1 米级的细节，但引擎却在为这些看不清的细节付全额代价。

多 LOD 的思路就是：远处用"大 block"来代表一片区域。同样一个 32³ 的 chunk，如果每个 block 代表 16 米（而不是 1 米），那这个 chunk 就覆盖 32×16 = 512 米见方的范围，顶点数却和近处 32 米的 chunk 差不多。于是世界从近到远用不同尺寸的 block：近处 1 米（精细），往外 2 米、4 米、8 米，最远 16 米（粗略）。同样大小的 chunk 容器，靠 block 实际尺寸变大来覆盖更远的范围，顶点数却随距离递减。

和普通模型 LOD 的本质区别：普通模型的 LOD 是"同一套网格，远处用减面后的版本"——几何体不变，三角面数递减。体素世界的 LOD 不一样：它不改网格拓扑，而是改 block 的实际尺寸。LOD0 的一个 block 是 1 立方米，LOD4 的一个 block 是 16 立方米（边长 16 米）。换句话说，远处不是"近处地形的简化版"，而是"用更粗的方块重新采样了一遍地形"。这是"用精度换性能"的 LOD 思路，但实现手段是"block 尺寸递增"而不是"同一套网格减面"。

所以体素大世界不是一层 MCWorld，而是5 层 MCWorld 叠在一起，由一个管理者统一持有。这个管理者就是 MCManager。本篇就讲它怎么把这 5 层组织起来、层与层之间怎么对齐。

2. 5 层结构：blockSize = 1 << i

2.1 MCManager 持有 5 个 MCWorld

MCManager 是体素引擎的主控，定义在 Assets\Scripts\Core\MCCore\Scene\MCManager.cs（namespace Assets.Scripts.MC，继承 SceneComponent）。它内部持有一个 MCWorld 数组，长度就是 5：

public static int MaxMCWorldLodLevel = 5;          // MCManager.cs:40

protected MCWorld[] m_Worlds;                       // MCManager.cs:42

public MCManager(UniversalScene scene) : base(scene)
{
    m_Worlds = new MCWorld[5];                      // MCManager.cs:115
    m_ChunkGameObjects = new Dictionary<YKVector3i, List<MCGameObjectBase>>[5];
    for (int i = 0; i < MaxMCWorldLodLevel; i++)
    {
        int blockSize = 1 << i;                    // 关键：blockSize 按 2 的幂递增
        m_Worlds[i] = new MCWorld(blockSize, 32);   // MCManager.cs:122  chunkSize 恒为 32
        m_ChunkGameObjects[i] = new Dictionary<YKVector3i, List<MCGameObjectBase>>();
    }
    ...
}

循环体只有一行是核心：m_Worlds[i] = new MCWorld(blockSize=1<<i, chunkSize=32)。第 i 层（LOD i）的 blockSize 就是 1 << i——位左移，等价于 2 的 i 次方。MCWorld 构造函数（Packages\com.unity.mccore\Runtime\MCWorld.cs:21）只是把这两个值存成只读字段：

public readonly float BlockSize = 1;
public readonly int ChunkSize = 32;

public MCWorld(float blockSize, int chunkSize)
{
    BlockSize = blockSize;
    ChunkSize = chunkSize;
    ...
}

所以 5 层 MCWorld 的结构完全一样（都是"字典里放一堆 32³ chunk"），唯一的差别就是 BlockSize 不同。这正是第 2 篇强调的"单层 MCWorld 的存储结构是一样的"——5 层共用同一套存储逻辑，只是 block 实际尺寸递增。

2.2 5 层参数表

把循环展开，5 层的参数一目了然：

LOD 层号blockSize (米)chunkSize (恒)单 chunk 覆盖边长 (米)典型用途
LOD 01 (= 1<<0)3232 × 1 = 32近处精细表现
LOD 12 (= 1<<1)3232 × 2 = 64中近距离
LOD 24 (= 1<<2)3232 × 4 = 128中距离
LOD 38 (= 1<<3)3232 × 8 = 256中远距离
LOD 416 (= 1<<4)3232 × 16 = 512远处粗略表现

"单 chunk 覆盖边长" = chunkSize × blockSize，也就是一个 chunk 在世界里实际占多大地。可以看到随着 LOD 升高，一个 chunk 覆盖的范围以 2 倍递增：32 → 64 → 128 → 256 → 512 米。同样 32³ 个方块，远处一个 chunk 顶 16 个近处 chunk 的覆盖面积（边长 4 倍，体积 64 倍）。

2.3 5 层同心结构

从玩家所在位置往外看，5 层是同心嵌套的：最内圈用 LOD0 精细方块，往外一圈切到 LOD1，再往外 LOD2、LOD3，最外圈是 LOD4 的粗略方块。哪一圈用哪一层，由 LOD 距离决定（见第 6 节），本节只看"叠起来长什么样"：

                     LOD4 (blockSize=16, chunk=512m)
          ┌──────────────────────────────────────────┐
          │              LOD3 (blockSize=8, chunk=256m)          │
          │     ┌────────────────────────────────┐     │
          │     │          LOD2 (blockSize=4, chunk=128m)      │     │
          │     │     ┌──────────────────────┐     │     │
          │     │     │    LOD1 (blockSize=2, chunk=64m)  │     │     │
          │     │     │   ┌──────────────┐   │     │     │
          │     │     │   │ LOD0 (1m, 32m)│   │     │     │
          │     │     │   │   ★ 玩家      │   │     │     │
          │     │     │   └──────────────┘   │     │     │
          │     │     └──────────────────────┘     │     │
          │     └────────────────────────────────┘     │
          └──────────────────────────────────────────┘

   越往外 block 越大、chunk 覆盖越广、顶点越少 → 用精度换性能

这张图是"概念示意"——实际不是规则的同心方框，而是由八叉树按距离动态切层的（第 4 篇展开扫描流程）。但分层关系就是这样：同一片世界空间，在 5 层 MCWorld 里各有一份"不同精度的采样"，近处用 LOD0 那份，远处用 LOD4 那份。

2.4 为什么 chunkSize 恒为 32、只让 blockSize 递增

这是一个值得讲清的设计选择。5 层里 chunkSize 写死 32，只有 blockSize 在变。为什么不让 chunk 也变大、或者干脆让高 LOD 用更稀疏的 chunk？

原因在于统一逻辑。chunk 是 mesh 构建和内存管理的基本单位：构网格时遍历一个 32³ 数组、邻居标脏时按 chunk 边界判定、对象池按 chunk 回收。如果 chunkSize 也变，那 mesh 构建循环、邻居边界检查、blockIndex 计算全得按 LOD 写成 5 套分支，维护成本很高。把 chunkSize 钉死在 32，意味着所有层的 chunk 内部逻辑完全一致——构网格、查 block、标脏、回收都共用同一套代码，只有"每个 block 代表几米"这一个参数在变。用 blockSize 一个参数调控覆盖范围，换来了 chunk 处理逻辑的完全统一。

一句话总结：chunk 是"逻辑单元"（mesh 构建、内存管理的粒度），尺寸恒定保证逻辑统一；block 是"采样粒度"（一个方块代表几米），尺寸递增实现远处粗略。两者职责分离，5 层只调后者。

3. 层间对齐：一拆八

5 层 MCWorld 各存各的 chunk，但它们描述的是同一片世界空间，只是精度不同。所以层与层之间必须能换算："高层这一个 chunk，对应低层哪几个 chunk？" 这是 LOD 体系里最基础的父子映射。

3.1 一拆八的几何关系

每降一层，blockSize 减半（16→8→4→2→1）。chunkSize 恒为 32，所以高层一个 chunk 的覆盖边长 = 32 × blockSize，低层一个 chunk 覆盖边长 = 32 × (blockSize/2) = 高层的一半。三维各减半，意味着高层一个 chunk 的空间，正好被低层 2×2×2 = 8 个 chunk 填满。每降一层就一拆八：

   LOD1 的一个 chunk (blockSize=2, 覆盖 64m)
   ┌──────────────────────────────┐
   │         │         │          │   ← 沿 X 拆成 2 份
   │  LOD0-0 │  LOD0-1 │          │
   │         │         │          │
   ├─────────┼─────────┤  ...     │   ← 沿 Y 拆成 2 份
   │         │         │          │
   │  LOD0-2 │  LOD0-3 │          │
   │         │         │          │
   └──────────────────────────────┘
        ↑
        沿 Z 还有一层（向纸内/外），共 2×2×2 = 8 个 LOD0 chunk

   LOD1 的 1 个 chunk  ──>  LOD0 的 8 个 chunk
   LOD2 的 1 个 chunk  ──>  LOD1 的 8 个  ──>  LOD0 的 64 个
   LOD3 的 1 个 chunk  ──>  LOD0 的 512 个
   LOD4 的 1 个 chunk  ──>  LOD0 的 4096 个

跨多层时就是连续一拆八：LOD2 的一个 chunk 对应 LOD0 的 8×8 = 64 个，LOD4 一个 chunk 对应 LOD0 的 8⁴ = 4096 个。这个 8 的幂次关系，正是 blockSize 按 2 的幂递增的直接几何后果。

3.2 GetChildChunkIndexList：算出子 chunk 的索引列表

把这个拆分关系写成代码的是 MCManager.GetChildChunkIndexList（MCManager.cs:251）。给定高层 LOD 的一个 chunkIndex，算出它在低层 LOD 对应的全部 chunkIndex：

// 根据 sourceLod 的 chunkIndex 获取对应 destLod 的 chunkIndex 列表
public void GetChildChunkIndexList(int sourceLod, int destLod, long chunkIndex,
                                   ref List<long> destChunkIndexList)
{
    if (destLod >= sourceLod)                       // destLod 必须更精细（更小）
    {
        Debug.LogError("destLod 必须小于 sourceLod !");
        return;
    }

    MCWorld world = m_Worlds[sourceLod];            // 取高层世界
    int blockSize = (int)world.BlockSize;           // 高层 blockSize（如 LOD2 = 4）
    int chunkSize = world.ChunkSize;                // 32
    int chunkWith = blockSize * chunkSize;          // 高层 chunk 实际边长（如 128）

    YKVector3d chunkId = NetWorkUtility.ToYKVector3_20(chunkIndex);
    YKVector3d chunkOffsetOrigin = chunkId * chunkWith;   // 高层 chunk 的世界坐标原点

    int destBlockSize = (int)Math.Pow(2, destLod);  // 低层 blockSize（如 LOD0 = 1）

    // 递归拆分：从高层的 blockSize 一路对半切到 destBlockSize
    GetLodChunksLoop((int)chunkOffsetOrigin.x, (int)chunkOffsetOrigin.y,
                     (int)chunkOffsetOrigin.z, chunkWith, blockSize,
                     destBlockSize, ref destChunkIndexList);
}

核心是递归函数 GetLodChunksLoop（MCManager.cs:222）。它每降一层就把当前空间沿 x/y/z 各切两份（共 8 份），对每一份继续递归，直到 blockSize 切到目标 destBlockSize 才把对应的 chunkIndex 收进结果列表：

private void GetLodChunksLoop(int x, int y, int z, int chunkSize, int blockSize,
                              int destBlockSize, ref List<long> mcChunks)
{
    int subBlockSize = blockSize / 2;               // 降一层：blockSize 减半
    int subChunkSize = chunkSize / 2;               // 对应 chunk 实际边长也减半
    float subChunkSizeInverse = 1.0f / subChunkSize;
    for (int w = 0; w < 2; w++)                     // 2×2×2 八方向
      for (int v = 0; v < 2; v++)
        for (int u = 0; u < 2; u++)
        {
            int offsetX = x + u * subChunkSize;     // 子 chunk 的世界坐标原点
            int offsetY = y + v * subChunkSize;
            int offsetZ = z + w * subChunkSize;

            if (destBlockSize < subBlockSize)       // 还没切到目标层，继续递归
                GetLodChunksLoop(offsetX, offsetY, offsetZ, subChunkSize,
                                 subBlockSize, destBlockSize, ref mcChunks);
            else if (destBlockSize == subBlockSize) // 切到目标层，收一个 chunkIndex
                mcChunks.Add(NetWorkUtility.ConvertXYZ_20(
                    (int)(offsetX * subChunkSizeInverse),
                    (int)(offsetY * subChunkSizeInverse),
                    (int)(offsetZ * subChunkSizeInverse)));
        }
}

三层嵌套 for（2×2×2）就是"一拆八"的代码化身；外层 if/else if 决定"继续往下拆"还是"到底了收结果"。如果 sourceLod 和 destLod 之间隔了好几层，递归就会层层往下拆，最终收进 8 的若干次方个 chunkIndex。

3.3 这个映射用来干嘛

父子映射在 LOD 体系里是"基础设施"，典型用途有两类：

  • 高层 chunk 数据变了，要通知低层刷新：远处 LOD4 的一个粗略 chunk 改了，它对应的 4096 个 LOD0 chunk 里那些还存在的，可能需要重新构网格。这时先用 GetChildChunkIndexList 算出子 chunk 列表，再逐个标脏。
  • 低层 chunk 要找对应的高层父块：反向查"我这个精细 chunk 归哪个粗略 chunk 管"，用于跨层引用、遮挡判定等。反向方向是除以 2 的幂、向下取整。

本篇只讲这个映射关系怎么算——即"高层 1 个 chunk 对应低层 8ⁿ 个 chunk"的几何和代码。至于"什么时候真的去触发跨层刷新、刷新走哪条调度路径"，那是 chunk 动态加载调度的事，留到第 4 篇展开。

3.4 blockSize 和 lod 一一对应：Math.Log 反算层号

因为 blockSize 严格按 1 << i 递增，blockSize 和 lod 是一一对应的：blockSize=1 ↔ lod=0，blockSize=2 ↔ lod=1，……，blockSize=16 ↔ lod=4。所以代码里经常看到用 blockSize 反算 lod 的写法：

int lod = (int)Mathf.Log(blockSize, 2);   // MCManager.cs:404 (SetBlock 里)
// 等价写法：(int)Math.Log(lod, 2)  —— 注意这里的形参 lod 其实是 blockSize

int index = (int)Mathf.Log(chunkEntity.BlockSize, 2);  // MCManager.cs:997 (AddChunk 里)
int index = (int)Math.Log(lod, 2);                     // MCManager.cs:1437, 1472

Math.Log(blockSize, 2) 就是"blockSize 是 2 的几次方"，结果正好是 lod 层号。这个写法在 MCManager 里反复出现（SetBlock、AddChunk、RemoveChunk、RemoveGameObject 都有），它把"blockSize 这个实际尺寸"和"lod 这个数组下标"两个表达统一起来：你给我一个 blockSize，我就能算出它在 m_Worlds[?] 的哪一层。

一一对应的代价：这个反算能成立，前提是 blockSize 严格是 2 的幂且和 lod 严格对应。一旦未来想插入一个 blockSize=3 的中间层，所有 Math.Log(blockSize, 2) 都会算出非整数、直接破。所以"5 层 + blockSize=1<<i"不只是一个初始化习惯，而是一个被全代码依赖的不变量。

4. 异形块 / Entity 只在 LOD0

5 层 LOD 不只是"block 尺寸递增"，还有一个容易忽略的分层约束：异形块（非标准立方体的装饰方块）只在 LOD0 实例化。这是"连内容类型都分层"的设计——远处不仅方块变大，连异形块这种额外内容都直接不画了。

4.1 BuildMeshEx 里的 BlockSize==1 判定

构网格的入口 BuildMeshEx（MCManager.cs:1144）在构完普通 mesh 后，有一段只对 LOD0 生效的额外逻辑：

public void BuildMeshEx(object obj)
{
    MCChunkRenderData data = (MCChunkRenderData)obj;
    ...
    if (bShow)
    {
        data.RenderWrap.LoadingStates = LoadingSessionStates.Finished;
        data.BuildMesh(_addGameObjectAction);          // 普通 mesh：所有层都构
        if (data.Chunk.BlockSize == 1)                 // 只有 LOD0 才做下面两件事
        {
            // 异形块实例化（走几何实例化的非标准方块）
            if (data.RenderWrap.IsEntityDirty)
            {
                data.BuildEntityInstance(_addEntityInstance);
            }
            // 装饰 entity 实例化
            Scene.ExecuteInThread(new UniversalScene.ThreadTask(data, _buildDecorationEntity));
        }
        data.RenderWrap.Source?.OnRenderTaskFinish(this);
    }
    ...
}

关键就是 if (data.Chunk.BlockSize == 1) 这一句。BlockSize == 1 就是 LOD0（见第 2 节参数表）。只有 LOD0 的 chunk 才会调用 BuildEntityInstance（异形块实例化）和 BuildDecorationEntity（装饰 entity 实例化）。LOD1~LOD4 的 chunk 只构普通 mesh，跳过这两步。新版渲染路径 BuildMeshExNew（MCManager.cs:1099）的判定完全一样：if (data.Chunk.BlockSize == 1)。

4.2 为什么只在 LOD0

异形块是"非标准立方体"的方块——斜坡、装饰物、特殊造型之类，通常走几何实例化（instancing）画出来。这些东西的实例化开销不小（要建 SceneEntity、要进渲染列表、要管生命周期），而且只有玩家凑近了才看得清造型。远处一个 LOD4 chunk 的 block 已经是 16 米一个的粗方块了，把里面的斜坡也用异形块画出来，既看不清也浪费——直接用一个粗略方块代表这一片就够了。

所以设计上做了取舍：异形块/装饰 entity 只在最近的 LOD0 实例化，LOD1 及以外统统不画。这样大部分远处 chunk 只构一份简单 mesh，省下大量实例化和 entity 管理开销。这是 LOD 体系里"内容也分层"的体现：不仅 block 尺寸递增，连"要不要画异形块"这件事也按层裁剪。

4.3 RemoveGameObject 里的对称判定 —— 命名陷阱

有构就有删。对称地，移除显示对象时也只在 LOD0 移除 entity。但这里的代码有一个非常容易踩坑的命名：

private bool RemoveGameObject(float lod, YKVector3i posIndex)
{
    if (lod == 1)                       // MCManager.cs:1529  ← 注意这个 lod 是 BlockSize！
    {
        RemoveChunkEntityObject(lod, posIndex);
    }
    return RemoveChunkBlockObject(lod, posIndex);
}

这个方法形参叫 lod，但传进来的实际是 chunk.BlockSize（见 MCManager.cs:1523：RemoveGameObject((int)chunk.BlockSize, chunk.Index)）。所以 if (lod == 1) 真正的意思是 "如果 BlockSize == 1"，也就是 LOD0。同样地，RemoveChunkEntityObject 内部（MCManager.cs:1477）的 if (index == 0) 才是真正的"lod == 0"——那里 index 是 (int)Math.Log(lod, 2) 算出来的真 lod。

更迷惑的是，BuildMeshEx 里那条注释写的是"只有Lod1的Chunk才会显示异型块"（MCManager.cs:1112, 1158），但代码判定的是 BlockSize == 1，对应的是 LOD0。

命名陷阱，务必当心：在这套代码里，"lod" 这个变量名有时指真正的 LOD 层号（0~4，用 Math.Log(blockSize, 2) 算出来），有时却被直接拿来装 BlockSize（1/2/4/8/16）。注释里说的"Lod1"指的是 BlockSize==1 即 LOD0，不是 LOD 层号 1。判定"是不是最精细层"时，BlockSize == 1、lod == 1（这里 lod 是 blockSize 形参）、index == 0（这里 index 是真 lod）三种写法都在用，说的却是同一件事。读这段代码时一定要先确认形参里那个 lod 到底是 blockSize 还是真层号，否则会把"LOD0 专属"误读成"LOD1 专属"。

这个陷阱的存在，本质上是因为 blockSize 和 lod 一一对应（见 3.4），早期写代码时直接拿 blockSize 当 lod 用、省一次 Math.Log，结果变量名就留下了歧义。读代码时记住"lod == 1 在这里 = BlockSize==1 = LOD0"就不会被注释带偏。

4.4 entity 的模型 LOD 是另一套系统

这里说的"异形块只在 LOD0"是chunk 地形/方块这一侧的事——只跟 5 层 MCWorld 有关。另外还有一套entity 自身的模型 LOD（同一个装饰物有 3 级精细模型，按距离切换），那是 SceneObjectManager 管的另一套系统，和这里的 chunk 地形 LOD 是分离的。本篇只讲地形/方块的 5 层 LOD，entity 模型 LOD 留到第 7 篇展开。

5. MCManager 还管什么（概览）

本篇把 MCManager 当作"5 层 MCWorld 的持有者"在讲，但它实际是体素引擎的主控，管的远不止 5 层世界。这里只做本篇范围内的概览，让你知道它还挂着哪些职责，但每项的具体调度细节都留到后面几篇。

5.1 MCManager 是体素引擎主控

MCManager 的类声明（MCManager.cs:22）挂了一长串接口：

public class MCManager : SceneComponent, ISchedulable, IThreadRunnable,
                         IRefreshable, IOnTerrainRangeChanged

可以看出它是个 SceneComponent（挂在场景上，第 1 篇讲过的场景组件机制），能被调度（ISchedulable）、能跑线程（IThreadRunnable）、能响应刷新（IRefreshable）、能响应画质档位变化（IOnTerrainRangeChanged，第 6 节会用到）。它手里攥着的体素相关资产，本篇范围内能点出来的有：

  • 5 层 MCWorld（m_Worlds）——本篇主角。
  • Patch 缓存 PatchesCache（MCManager.cs:148）——地块（MCPatch）的容器，chunk 的动态加载/卸载都围绕 Patch 的八叉树展开。本篇不展开，第 4、5 篇详讲。
  • SetBlock（MCManager.cs:402）——改方块入口，内部按 blockSize 算出 lod，再落到对应 MCWorld 的 chunk。
  • 邻居刷新 CheckRefreshNeighbors（MCManager.cs:857）——block 改在 chunk 边界时，要给相邻 chunk 标脏。
  • mesh 构建调度 RefreshChunk / RendererChunk / BuildMeshEx——把脏 chunk 推给图形线程构网格。调度流程归第 4 篇。
  • CurrentMCWorld 属性——常用访问入口，下面单独讲。

5.2 GetMcWorld(lod)：按层取用

既然 MCManager 持有 5 层 MCWorld，外部自然要能按层取用。取用接口就是 GetMcWorld（MCManager.cs:171），非常直白：

public MCWorld GetMcWorld(int lod)
{
    return m_Worlds[lod];
}

给个 lod 层号（0~4），返回对应的 MCWorld。WorldSceneSystem 里就有按指定 lod 查 chunk 的用法（WorldSceneSystem.cs:1058, 1072, 1083, 1090, 1099 一带），都是 MCScene.MCManager.GetMcWorld(lod).TryGetChunk(...) 这种套路——想查非 LOD0 的 chunk，就显式传 lod。

5.3 CurrentMCWorld：默认走 LOD0

大多数业务查方块不需要指定 lod，因为玩家日常交互（点击、放置、射线检测）都发生在玩家附近的精细区域，也就是 LOD0。MCManager 暴露了一个快捷属性 CurrentMCWorld（MCManager.cs:108），固定指向 LOD0：

public MCWorld CurrentMCWorld => m_Worlds[0];   // 永远是 LOD0

于是最常见的"查一个方块"访问路径就是 MCScene.MCManager.CurrentMCWorld.TryGetBlock。WorldSceneSystem 的 TryGetBlock（WorldSceneSystem.cs:561）就是这么走的：

public bool TryGetBlock(float x, float y, float z, out MCBlock block)
{
    if (MCScene == null || MCScene.MCManager == null
        || MCScene.MCManager.CurrentMCWorld == null) { ... }
    if (MCScene.MCManager.CurrentMCWorld.TryGetBlock(x, y, z, out block))   // ← 走 LOD0
    { ... }
    ...
}

这条路径你已经在第 2 篇末尾见过了——WorldSceneSystem.TryGetBlock 内部就是调 CurrentMCWorld.TryGetBlock，而 CurrentMCWorld 就是 m_Worlds[0]。把这条路径记住：默认查询都走 LOD0，要查别的层才显式 GetMcWorld(lod)。

本篇边界：MCManager 作为"5 层 MCWorld 持有者"和"按层取用入口"这两个角色，是本篇讲清的内容。Patch 缓存怎么用、mesh 构建怎么调度、chunk 怎么动态加载卸载，都是建立在 5 层结构之上的调度层话题，留到第 4 篇起展开。

6. LOD 距离的动态调整（概览）

第 2 节那张同心结构图是"概念示意"，真实世界里哪一圈用哪层 LOD，是由每层的查询距离决定的——"离玩家多近才用 LOD0、多远才切到 LOD1"这些阈值不是写死的，而是可调的。本节只点出"这套不是写死的、有画质档位、有特殊场景调整"，让你知道有这回事；具体 Patch 怎么拿这些距离去做加载判定，留到第 4、5 篇。

6.1 OctreeQueryByLod 与 _lodDis

LOD 距离的具体载体是 OctreeQueryByLod（定义在 Assets\Scripts\Core\MCCore\Octree\Octree.cs:650），它核心持有两个东西：

public class OctreeQueryByLod
{
    public YKVector3d _pos;   // 观察位置（查询中心，一般是玩家位置）
    public float[] _lodDis;   // 各层 LOD 的查询距离阈值（按八叉树深度索引）
    public bool Contain(...)  // 判定一个节点中心是否落在某层 LOD 范围内
    ...
}

_lodDis 是一个 float 数组，存的是"各层 LOD 的查询距离"。每个 MCPatch 持有一个这样的查询器（MCPatch.m_Query，MCPatch.cs:258），MCManager 自己也缓存了一份（MCManager.m_PatchQuery，MCManager.cs:111，在 OnPatchFill 里从 patch 复制过来）。

数组下标的小坑：_lodDis 的下标是八叉树深度，不是直接是 LOD 层号。从 MCPatch 里的注释（MCPatch.cs:826-849）可以看出，下标 4~8 对应 blockSize 16/8/4/2/1，也就是 LOD 4/3/2/1/0（下标越大、层越精细、距离阈值越小）。下标 0~3 是给八叉树上层大节点用的占位（统一给 8000 这种大值，相当于"永远在范围内"）。所以读 _lodDis[8] 要知道它说的是 LOD0 的距离，不是第 8 层 LOD。

6.2 画质档位缩放：OnTerrainRangeChanged

LOD 距离会跟着画质档位缩放。MCManager 实现了 IOnTerrainRangeChanged，对应方法 OnTerrainRangeChanged（MCManager.cs:1796）：

public virtual void OnTerrainRangeChanged()
{
    TerrainConfig config = Scene.SceneConfig.terrainInfo;
    m_TerrainRange = UniversalRenderPipeline.asset.PipelineRenderSettingData.TerrainRange;
    Vector2 scheduleRange = new Vector2(config.BlockConfig.XLoadRange, config.BlockConfig.ZLoadRange);
    switch (m_TerrainRange)
    {
        case LowMemory:
        case Low:        scheduleRange *= 0.5f;  break;   // 低画质：加载范围砍半
        case Medium:
        case High:       scheduleRange *= 1.0f;  break;   // 中/高：基准
        case Ultra:      scheduleRange *= 2.0f;  break;   // 极高：翻倍
    }
    ...
    // 同步把每层 LOD 的查询距离更新到各 patch
    foreach (var it in PatchesCache.Patches)
        ((MCPatch)it.Value).SetQueryLodDistance(m_TerrainRange);
}

这里有 5 个档位：Low / LowMemory / Medium / High / Ultra。ScheduleRange（整体调度范围）按档位 0.5x / 1x / 2x 缩放，同时调用每个 patch 的 SetQueryLodDistance 重设 _lodDis。SetQueryLodDistance（MCPatch.cs:857）内部也是按这 5 个档位给 _lodDis[4~8] 填不同的距离值——比如 Ultra 档 LOD0(_lodDis[8]) 给 128，Low 档只给 96，意味着低画质下玩家更近才会加载 LOD0 精细块。

6.3 特殊场景的 LOD 调整

除了画质档位，还有两种针对特殊场景的 LOD 微调入口：

  • EnlargeLod0Distance（MCPatch.cs:979）：编辑器单机模式下，把 LOD0 的可见距离放大（_lodDis[8] 从 96 提到 512），让美术在编辑器里能看到更远处的精细地形。
  • SetSkiBattleLodDistance（MCScene.cs:1040，内部调 MCPatch.SkiBattleLodSetting MCPatch.cs:966）：滑雪单局场景专用的一组 LOD 距离，因为滑雪玩法视角和移动速度跟主城不一样，需要单独调一套阈值。调用方在 LuSkiSceneSystem.cs:43。

这两个入口都是直接改 _lodDis 数组的几个槽位，本质上就是"换一套距离阈值"。本篇只要知道：LOD 距离不是常量，有画质档位统一缩放、有特殊场景单独覆盖。至于 patch 怎么用这些距离、在八叉树上怎么扫描出该加载哪些 chunk，是第 4、5 篇的调度话题。

7. 小结与下一站

本篇把体素大世界的分层骨架讲清了，核心就三件事：

  • 5 层 MCWorld，blockSize = 1 << i：MCManager 持有 m_Worlds[5]，第 i 层 blockSize 是 2 的 i 次方（1/2/4/8/16），chunkSize 恒为 32。chunk 是"逻辑单元"尺寸恒定保证处理逻辑统一，block 是"采样粒度"尺寸递增实现远处粗略。这是"用精度换性能"的体素 LOD，和普通模型 LOD 的本质区别在于"改 block 尺寸"而不是"减面"。
  • 层间对齐靠一拆八：每降一层 blockSize 减半，三维各减半 → 一个高层 chunk 对应低层 2×2×2 = 8 个。跨多层就是 8 的幂次。GetChildChunkIndexList + GetLodChunksLoop 用递归三层 for 把这个拆分关系算出来。blockSize 和 lod 一一对应，所以代码里到处用 Math.Log(blockSize, 2) 反算层号。
  • 异形块只在 LOD0：BuildMeshEx 里 BlockSize == 1 才实例化异形块/装饰 entity，LOD1 及以上不画，省下大量实例化开销。注意 RemoveGameObject 里的命名陷阱——形参 lod 实际是 BlockSize，lod == 1 指 LOD0，注释里"Lod1"也是指 BlockSize==1，别被带偏。

另外点出了两个概览：MCManager 是体素引擎主控，本篇只讲它"5 层 MCWorld 持有者"的角色，GetMcWorld(lod) 按层取用、CurrentMCWorld（永远 LOD0）是默认查询入口（WorldSceneSystem.TryGetBlock 就走这条）；LOD 距离不是写死的，靠 OctreeQueryByLod._lodDis 存阈值，有 5 档画质缩放和滑雪等特殊场景覆盖。

到这里，体素大世界的静态分层结构就讲完了——5 层世界怎么叠、怎么对齐、各自存什么。但有一个问题一直没回答：这 5 层的 chunk 是什么时候被创建出来、什么时候被回收？玩家走动时，近处的 LOD0 chunk 怎么随着距离动态出现、远处的又怎么消失？这就是第 4 篇要讲的 Chunk 动态加载与释放——把这套静态骨架变成"跟着玩家跑的动态世界"。
