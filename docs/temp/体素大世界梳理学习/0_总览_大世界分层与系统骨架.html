总览 —— 大世界分层与系统骨架

这一篇是地图，不是工程手册：先看清整个体素大世界是怎么分层的、靠什么机制驱动、主角是怎么进来的，再逐篇深入各子系统。

在开始拆细节之前，先回答三个问题：大世界的代码分成哪几层、各层之间靠什么通信、玩家从点进游戏到看见主角站在世界上经历了哪条主链路。把这三件事在脑子里搭好框架，后面 11 篇每读一篇都能挂回这张地图上。

1. 三层架构总图

整个体素大世界在工程上被切成三层，从下往上依次是：引擎场景底座、MC 体素引擎、业务系统层。每一层只对上一层暴露少量入口，对下一层只依赖明确的接口，这种分层是后面所有内容的地基。

┌─────────────────────────────────────────────────────────────────────────────┐ │ 上层 · 业务系统层 (GameWorld.System) │ │ WorldSceneSystem ──统一对外封装 MCScene 接口（SetBlock / TryGetBlock / SetEntity …）│ │ │ 兄弟系统：WorldNetSystem / BattlePlayerSystem / CameraSystem / │ │ │ ObjectConfigSystem / LuEntityVoxelSystem / ChunkSystem … │ │ ▼ 对下：调用 m_McScene（MCScene）/ m_MCNetManager（MCManager） │ ├─────────────────────────────────────────────────────────────────────────────┤ │ 中层 · MC 体素引擎 │ │ Assets.Scripts.MC MCScene / MCManager / MCNetworkManager / MCPlaceholderManager│ │ │ MCManager 内部挂 5 层 MCWorld（m_Worlds[5]，blockSize=1<<i，chunkSize=32）│ │ ▼ 对下：把 MCWorld / MCChunk / MCBlock 作为体素数据原语 │ │ MCCore (Packages) MCWorld / MCChunk / MCBlock —— 纯体素数据结构 │ ├─────────────────────────────────────────────────────────────────────────────┤ │ 底层 · 引擎场景框架底座 (UnityEngine.Scene.Universal) │ │ UniversalScene / UniversalSceneManager / SceneObjectManager / SceneComponent │ │ 职责：场景生命周期、加载回调、SceneObject 管理、组件化调度 │ │ 对上：UniversalSceneManager.CreateScene<T>(...) 工厂方法创建场景 │ └─────────────────────────────────────────────────────────────────────────────┘

1.1 三层各自的“职责与边界”

层代表类对上提供什么对下依赖什么
业务系统层 WorldSceneSystem 等 给上层（含 Lua）一套业务语义接口：放块、取块、摆物件、跳转坐标、loading 进度 调用 MCScene / MCManager；通过 Owner.GetSystem<T>() 拿兄弟系统
MC 体素引擎 MCScene / MCManager / MCWorld 体素世界的数据与流程：chunk 的加载/卸载、block 的读写、LOD 多世界、物件占位 继承自底层的 UniversalScene；用 SceneObjectManager 管理物件
引擎场景底座 UniversalSceneManager / UniversalScene 场景工厂 CreateScene<T>、加载完成回调、SceneObject 增删查、组件化 不依赖上层，是整个体系的“地基”

关键的一点：业务层从来不直接 new 场景，也不直接碰 MCWorld/Chunk/Block 的原始结构。它通过 UniversalSceneManager.CreateScene<MCScene>(...) 拿到一个 MCScene 实例，之后所有操作都走 WorldSceneSystem 这一层封装。WorldSceneSystem.cs:176 这一行就是业务层“踩进”引擎层的入口：

m_McScene = UniversalSceneManager.CreateScene<MCScene>(path, scene => { OnLoadingFinishAction(); }, true, isNetManager);
为什么要分三层？底层 UniversalScene 是通用的“场景 + 物件 + 组件”框架，不绑定体素语义；中层的 MC* 在它之上叠出“按 chunk 切分、按 block 寻址、按 LOD 分辨率”的体素世界；上层再把这套东西包装成业务能直接调的接口。这样体素数据结构可以单独演进，业务逻辑也不必关心 chunk 怎么切。

1.2 各层源码位置一览

  • 底层：Assets\Scripts\Engine\UniversalScene\（UniversalScene.cs / UniversalSceneManager.cs，namespace UnityEngine.Scene.Universal）
  • 中层（场景侧）：Assets\Scripts\Core\MCCore\Scene\（MCScene.cs / MCManager.cs / MCNetworkManager.cs，namespace Assets.Scripts.MC）
  • 中层（体素数据）：Packages\com.unity.mccore\Runtime\（MCWorld.cs / MCChunk / MCBlock，namespace MCCore）
  • 上层：Assets\Scripts\Game\GameWorld\Game\OpenWorldGameSystem\（WorldSceneSystem.cs，namespace GameWorld.System）

各层内部如何组织、关键方法做什么，分别在第 1 篇（底层）、第 2 篇（体素数据）、第 3 篇（MCManager）展开。本篇只标位置，不展开实现。

2. IGameSystem 框架与事件总线

业务系统层不是一个松散的目录，而是一个被统一驱动的小框架。所有业务系统都继承自 IGameSystem，被宿主 GameBase 持有一份 list，按统一的生命周期被驱动；系统之间不直接互调，而是通过事件总线发布/订阅消息。

2.1 IGameSystem 的生命周期

IGameSystem 是个抽象类（不是接口），构造时把宿主 GameBase 作为 Owner 注入进来。IGameSystem.cs:27 提供了五个生命周期钩子，都是 internal virtual，子类按需 override：

  • Initialize()：初始化，拿兄弟系统引用、订阅事件
  • Update() / LateUpdate() / FixedUpdate()：每帧驱动
  • Shutdown()：卸载，反向清理（先加的后卸）
  • ReLogin(callback) / ReLoginComplete()：断线重连专用

GameBase 用一个 List<IGameSystem> + Dictionary<Type, IGameSystem> 同时持有所有系统，前者保证遍历顺序、后者保证按类型 O(1) 取用。GameBase.cs:36。所有系统通过 AddSystem<T>() 注册，GetSystem<T>() 拿兄弟系统：

// GameBase 统一驱动所有 System
foreach (var system in m_GameSystems) system.Initialize();   // 正向初始化
foreach (var system in m_GameSystems) system.Update();        // 每帧
for (int i = m_GameSystems.Count-1; i >= 0; i--) system.Shutdown(); // 反向卸载

WorldSceneSystem 就是个典型例子：它在 Initialize() 里通过 Owner.GetSystem<...>() 把 CameraSystem、ObjectConfigSystem 等兄弟系统拿过来缓存，再订阅自己关心的事件。WorldSceneSystem.cs:85。

设计要点：系统之间不互相持有，只通过 Owner.GetSystem<T>() 拿引用；事件不直接回调，只通过 Owner.Subscribe / Owner.FireNow。这样加一个新系统、或换一个系统的实现，都不需要改别的系统。

2.2 事件总线：Subscribe / FireNow

GameBase 内部持有一个 EventManager（来自 com.unity.youkiaenginecore 包），对外暴露三个方法：GameBase.cs:192

  • Subscribe(int evtID, EventHandler<GameEventArgs> handler)：按事件 ID 订阅
  • Unsubscribe(int evtID, handler)：取消订阅
  • FireNow(object sender, GameEventArgs e)：立即同步派发（注意不是下一帧）

EventManager 内部是个 EventPool<GameEventArgs>，模式为 AllowNoHandler | AllowMultiHandler——一个事件可以没人订阅，也可以被多个系统订阅，都不会报错。EventManager.cs:14。这正是“主角入场”这类事件能被多个系统同时关心的基础。

2.3 事件 Args 的写法与对象池复用

每一个具体事件都在 Assets\Scripts\Game\GameWorld\Game\Events\ 下定义（namespace GameWorld.Events），继承自 GameEventArgs，写法高度统一。以 WorldSceneSystemInitSceneArgs 为例 WorldSceneSystemInitSceneArgs.cs:15：

public class WorldSceneSystemInitSceneArgs : GameEventArgs
{
    public static readonly int EventID = typeof(WorldSceneSystemInitSceneArgs).GetHashCode();
    public override int Id => EventID;
    public override void Clear() { }
    public static WorldSceneSystemInitSceneArgs Create()
    {
        return ReferencePool.Acquire<WorldSceneSystemInitSceneArgs>();
    }
}

三个固定套路：

  1. EventID：用类型本身的 GetHashCode() 当事件 ID，保证全局唯一、不用手维护枚举。订阅和派发都引用同一个 EventID 字段。
  2. Create()：不 new，而是从 ReferencePool.Acquire<T>() 取一个复用对象（避免每次派发事件都产生 GC）。
  3. Clear()：对象归还池子前重置字段，由 ReferencePool 框架在 Release 时回调。

带数据的事件多一个字段 + 在 Create 里赋值 + 在 Clear 里置空。比如 WorldSceneSystemSetMainPlayerArgs WorldSceneSystemSetMainPlayerArgs.cs:28 在 Create(BattleRole role) 时把主角塞进去，订阅方 (e as XxxArgs)?.MainPlayer 取出来用。

3. 从启动到 LoadScene 的进入时序

这是本篇最重要的一节：把“玩家点进大世界 → 主角站在地上”这条主链路一次性画出来。细节留给第 5 篇，这里只让你拿到“全局路线感”。

3.1 WorldSceneSystem 怎么被初始化

WorldSceneSystem.Initialize() WorldSceneSystem.cs:85 做了四件事，为后续 LoadScene 铺好上下文：

  1. 通过 Owner.GetSystem<T>() 拿到 5 个兄弟系统引用并缓存（Camera、ObjectConfig、McEntityTimeState、ClientScanner、LuEntityVoxel）。
  2. 从相机系统取 SceneCamera，后续扫描中心点要用。
  3. Owner.Subscribe(WorldSceneSystemSetMainPlayerArgs.EventID, OnWorldSceneSystemSetMainPlayer)——订阅“主角装配完成”事件，这是进入链路最后一环的入口。
  4. EntityInjectUtility.Init()——初始化物件注入工具。

3.2 LoadScene 主链路时序图

当主角坐标从服务器拿到后，外部调用 WorldSceneSystem.LoadScene(path, px, py, pz, isNet) WorldSceneSystem.cs:171，之后链路自动推进到主角入场。整条链如下：

外部触发 WorldSceneSystem.LoadScene(path, px, py, pz, isNet) WorldSceneSystem.cs:171 │ ├─① ChangeScanTypeToLocation(px,py,pz) 把扫描中心先钉在“登录坐标” ├─② UniversalSceneManager.CreateScene<MCScene>(path, OnLoadingFinishAction, true, isNet) │ 底层工厂创建 MCScene；加载完成回调 = OnLoadingFinishAction ├─③ MCScene.LoadingType = LoadingType.Login 标记这是“登录式”加载 ├─④ m_MCNetManager = m_McScene.MCManager 拿到中层 MCManager 引用 ├─⑤ MCScene.SetLodInfo(m_LodInfo) / SetScanCenterLocation() │ 把相机视锥/扫描中心交给引擎 ├─⑥ Owner.FireNow(WorldSceneSystemInitSceneArgs) 广播“场景已创建” ├─⑦ 注册 GameWorkChunkPressureSystem / GameWorldServerChunkEntitySystem │ 让两个兄弟系统挂上 chunk 压力 / 服务器 chunk 物件 回调 ▼ 引擎加载完成回调 OnLoadingFinishAction() ──> CheckLoadingFinish() WorldSceneSystem.cs:242 │ ├─⑧ sceneFinished = MCScene.IsLoadingStateFinish() 引擎自报“数据都到齐了吗” │ 是 → 进度推到 SCENELOADFINISH(85%) │ Owner.FireNow(WorldSceneSystemMcSceneLoadFinishArgs) │ WorldNetSystem.SendJoinScene(null) 请求服务器“主角入场” │ 否 → 留在 loading，等下一帧 Update 再 Check ▼ 服务器回：主角装配完成 BattlePlayerSystem.MainRoleInitFinish() │ Owner.FireNow(WorldSceneSystemSetMainPlayerArgs.Create(m_MainRole)) BattlePlayerSystem.cs:471 ▼ WorldSceneSystem 收到事件 OnWorldSceneSystemSetMainPlayer(e) WorldSceneSystem.cs:105 │ m_MainPlayer = args.MainPlayer ▼ 主角入场收尾 MainPlayerJoin() WorldSceneSystem.cs:219 ├─⑨ ChangeScanTypeToRole() 扫描中心从“登录坐标”切到“跟着主角” ├─⑩ IsLoadingFinish = true；开海平面；FireNow(WorldSceneSystemRenderSceneArgs) └─⑪ ManualLoadingProgress(CLOSELOADING) → CsCallLuaBridge.SceneLoadFinish() 通知 Lua 关掉 loading UI

3.3 每一步在干什么（一句话版）

  • LoadScene：把扫描中心钉在登录坐标，用底层工厂造出 MCScene，标记登录式加载，把 LOD/扫描信息喂给引擎，广播“场景已建好”，并把两个 chunk 相关兄弟系统挂上去。
  • OnLoadingFinishAction → CheckLoadingFinish：引擎把玩家附近的 chunk 数据加载完会回调这里，这里再问一次引擎“真的都到齐了吗”，到齐就推 loading 进度、广播完成事件、向服务器请求主角入场。
  • SendJoinScene：通过 WorldNetSystem 走 Lua 桥让服务器装配主角；主角装好后由 BattlePlayerSystem 派发 SetMainPlayer 事件。
  • OnWorldSceneSystemSetMainPlayer：WorldSceneSystem 收到事件，把主角存下来，调 MainPlayerJoin。
  • MainPlayerJoin：扫描中心切到“跟着主角跑”，置完成标志，广播“场景可渲染”，把 loading 进度推到关 UI 档位，通知 Lua 收尾。
这里有个刻意设计：链路不是“场景加载”和“主角装配”并行，而是先等场景数据完全到齐，再请求主角入场（代码注释也写明了这个方案，见 WorldSceneSystem.cs:215）。原因是为了把 CPU 先让给场景逻辑，避免主角数据和 chunk 数据抢资源导致卡顿。注释里也提到后续可以改成并行。这个设计的来龙去脉和各判定条件会在第 5 篇和第 9 篇展开。
