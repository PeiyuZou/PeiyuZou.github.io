## 一、为什么需要引用计数？

`AssetBundle.Unload(true)` 会将 AB 以及 AB 加载出来的 Asset 对象全部卸载掉。但 Asset 对象很可能还在被使用，导致游戏异常。所以必须必须维护 "这个 AB 还有多少地方在用"，归零才能释放。引用计数就是干这个事的。

## 二、ResourceRef

每个使用 AB 资源的 GameObject 都挂一个 ResourceRef（或其子类 SpriteRef / MaterialRef 等）。当 GameObject 被销毁，OnDestroy 自动 ReduceRefCount。

```cs title="ResourceRef"
public class ResourceRef : MonoBehaviour
{
    [SerializeField] private string mAbPath;
    [SerializeField] private string mAssetName;
    [SerializeField] private IAwakeRef ar;
    [SerializeField] private int instanceID;

    public virtual void OnDestroy()
    {
        if (!string.IsNullOrEmpty(mAbPath))
        {
            ResourceManager.Instance.ReduceRefCount(mAbPath, mAssetName);
            mAbPath = null;
            mAssetName = null;
        }
        ar = null;
    }
}
```

除此之外，还需要按照资源类型进行派生空类。比如，一个 Button 可能同时引用一张图（SpriteRef）和一个材质（MaterialRef），两个引用必须能并存。所以有以下派生：

```cs title="空类派生"
public class ObjectRef    : ResourceRef {}
public class TextureRef   : ResourceRef {}
public class MaterialRef  : ResourceRef {}
public class SpriteRef    : ResourceRef {}
```

## 三、CheckClone，防止克隆导致计数错乱

```cs title="CheckClone代码"
public void CheckClone()
{
    if (instanceID == 0)
    {
        instanceID = gameObject.GetInstanceID();
    }
    else
    {
        int thisInstanceID = gameObject.GetInstanceID();
        bool cloned = instanceID != thisInstanceID;
        instanceID = thisInstanceID;

        if (cloned)
        {
            if (!string.IsNullOrEmpty(mAbPath))
                ResourceManager.Instance.AddRefCount(mAbPath, mAssetName);
            ar?.AddSleepy(this);
        }
    }
}
```

GameObject.Instantiate 复制对象时，组件上的所有字段都会被复制，包括 mAbPath、mAssetName。但被复制出来的新 ResourceRef 从未调用过 AddRefCount，它的引用是"白嫖"来的。更糟的是：当新对象 OnDestroy 时，它会 ReduceRefCount，引用计数会被错减，导致 AB 提前卸载。

序列化的字段保留着原始对象的 instanceID，一旦发现"我现在的 instanceID 跟字段里存的不一样"，就说明我是被克隆出来的。这时主动 AddRefCount 补一笔，让计数对账平。

```cs title="时序图"
[原始 Prefab 上挂着 ResourceRef]
   字段：mAbPath="ui/x"
         instanceID=0     ← 序列化时是 0
            │
            ▼
   AddRefFile(gameObject, "ui/x", "icon")
      ├─ CheckClone()
      │    instanceID==0 → 写入 thisInstanceID（比如 1000），不加引用
      └─ ReplaceResource("ui/x", "icon") → AddRefCount   ✓

[Instantiate 一次]
   新对象的字段被深拷贝：
         mAbPath="ui/x"
         instanceID=1000  ← 同步过来的，不是新对象自己的
            │
            ▼
   外部再次调 AddRefFile(newGo, "ui/x", "icon")
      ├─ CheckClone()
      │    thisInstanceID=2000，跟字段里的 1000 不同 → 克隆识别成立
      │    instanceID = 2000
      │    AddRefCount("ui/x", "icon")    ✓ 补上一笔
      └─ ReplaceResource(...) 路径相同：直接返回，不再 AddRef
```

## 四、AwakeRef：未激活对象的"延迟引用"

考虑一种情况：对象池里的 Prefab 对象，很多一开始是 SetActive(false) 的。
Unity 的 Awake 只在第一次激活时调用。这意味着如果你把引用计数登记放在 ResourceRef.Awake 里，未激活的对象永远不会登记，归零卸载就崩了。

解法是引入一个 "睡眠登记" 机制：

```cs title="AwakeRef"
public class AwakeRef : MonoBehaviour, IAwakeRef
{
    private HashSet<ResourceRef> SleepyRefs = new HashSet<ResourceRef>();

    void OnDestroy()
    {
        foreach (ResourceRef sleepyRef in SleepyRefs)
            sleepyRef.OnDestroy();   // ← 帮还没醒来的人代偿
        SleepyRefs.Clear();
    }

    public void AddSleepy   (ResourceRef rr) { SleepyRefs.Add(rr); }
    public void RemoveSleepy(ResourceRef rr) { SleepyRefs.Remove(rr); }
}
```

```cs title="ResourceRef登记"
void Awake()
{
    if (ar != null && !string.IsNullOrEmpty(mAbPath))
    {
        ar.RemoveSleepy(this);   // 我醒了，不需要别人代我还
        ar = null;
    }
}

public void FallSleep(IAwakeRef _ar)
{
    ar = _ar;
    ar.AddSleepy(this);          // 我睡了，让上层代我管
}
```

``` title="工作模型"
[Prefab 根节点挂 AwakeRef]
    │
    ├─ 子节点 A (active=true)   挂 ResourceRef
    └─ 子节点 B (active=false)  挂 ResourceRef → FallSleep(root.AwakeRef)
                                              ar.AddSleepy(B)

加载完成后：
    A.Awake() 立刻被调，A 自管自的引用。
    B 一直没醒过来，但被登记在 root.AwakeRef.SleepyRefs 里。

如果整个 Prefab 被销毁（玩家关掉这个面板）：
    root.AwakeRef.OnDestroy()
        foreach (b in SleepyRefs) b.OnDestroy();   // 代为还引用
    A.OnDestroy()                                  // 自己还自己的

如果 B 在中途被激活：
    B.Awake() → ar.RemoveSleepy(this), ar = null    // 从此独立
```

## 五、踩坑点-忘记挂 ResourceRef

如果你 ResourceManager.LoadObject("xxx", cb) 拿到 Prefab，Instantiate 出来直接用，但没有挂 ResourceRef，那么这个 AB 永远不会被引用计数管控。
