## xLua 概述

Unity 使用 C# 作为用户编程语言，为了解决热更新的问题，使用 Lua 是一种方案，xLua 解决的其中一个主要问题，就是 C# 和 Lua 的互相调用。

xLua 实现跨域调用，核心解决的三件事是：

- 把 C# 的对象/值，压栈（Push）到 Lua 栈上
- 把 Lua 栈上的值 取下来（Get / Cast）转换为 C# 类型
- 跨越 native（C 写的 Lua 内核）/ managed（C#）边界，依赖 P/Invoke

简单来讲，C# 通过 P/Invoke 可以访问并调用 Lua 提供的一些 API，由此可以对 Lua 栈进行一些操作，比如将指定的 Lua 函数压栈，再将一些需要的参数压栈，执行完成后，将栈上的执行结果抓回来，转换成 C# 的类型。这就是 C# 调用 Lua 的一个基本流程，这里容易产生一些疑问：

- 为什么 Lua 和 C# 无法直接互操作？
- 为什么选择 Lua 虚拟机栈作为跨域调用的核心角色？而不是选 CLR 的调用栈？
- P/Invoke 是什么？它的工作原理？

## 为什么 Lua 和 C# 无法直接互操作？

Lua 运行在 Lua VM 上，C# 运行在 CLR（Mono 或 IL2CPP）上，两者属于不同的运行时环境，默认情况下无法直接互相调用，需要借助 xLua 等桥接层完成通信。Lua 对象（Table、Closure、String 等）的内存由 Lua GC 管理，Lua 内核由 C 编写，本质上属于 Native Memory；而 C# 普通对象的内存由 Mono/IL2CPP GC 管理，属于 Managed Memory。不过 UnityEngine.Object（如 Texture、Mesh、GameObject）虽然由 C# 创建和访问，其实际数据主要存储在 Unity Native Memory 中。

``` title="Unity进程的内存划分"
Unity.exe 进程
│
├─ CLR/Mono/IL2CPP Managed Heap
│   ├─ C#对象
│   ├─ string
│   ├─ List<T>
│   └─ Delegate
│
├─ Unity Native Memory
│   ├─ Texture
│   ├─ Mesh
│   ├─ AudioClip
│   ├─ GameObject
│   ├─ Transform
│   └─ AssetBundle
│
├─ Lua VM Memory（lua_State）
│   ├─ Lua Table
│   ├─ Lua String
│   ├─ Lua Function
│   ├─ Lua Thread(coroutine)
│   ├─ Lua Userdata
│   └─ GC Object
│
└─ 其它Native库
    ├─ PhysX
    ├─ FMOD
    └─ ...
```

!!! note "这些内存区可能是交错存在的"
    Lua 不会向系统申请一个类似 `0x10000000 ~ 0x1FFFFFFF` 的连续的专门的 Lua 区域，而是每次从进程堆中拿内存，比如 `0x1000` 是一个 Lua Table，`0x2000` 是一个 C# Object，`0x3000` 又是一个 Texture Metadata

## 为什么选择 Lua 虚拟机栈？

Lua 使用的是 Stack-Based VM，而不是 Register-Based API，也就是说它直接提供了大量的操作栈的接口，比如：

```lua
lua_pushnumber(L, 123);

lua_pushstring(L, "hello");

lua_call(L, 1, 1);

lua_tonumber(L, -1);
```

可以说，Lua VM 直接将它的调用栈暴露给外部，作为数据交换通道，天然支持别的语言来交换数据。而 CLR 栈是不公开的，无法通过它交换数据来互操作。

## P/Invoke 是什么？它的工作原理？

P/Invoke 全称是 Platform Invocation（平台调用），它是 .NET 提供的可用于从托管代码访问非托管库中的结构、回调和函数的一种技术。

大多数 P/Invoke API 包含在以下两个命名空间中：System 和 System.Runtime.InteropServices。使用这两个命名空间可提供用于描述如何与本机组件通信的工具。

来看一个示例，下方是 xLua 的 `LuaDLL.cs` 脚本中定义的一个可跨域调用的 xLua native 函数：

```cs
[DllImport("xlua", CallingConvention = CallingConvention.Cdecl)]
public static extern IntPtr lua_tothread(IntPtr L, int index);
```

从以下几个方面看：

### 1. extern 关键字

这个关键字表示：这个函数的实现不在 C# 代码里，而是在外部。CLR 看到 `extern` 后会知道：这个函数需要去 Native DLL 里找实现。

### 2. DllImport 特性

它告诉 CLR，该函数的实现位于 `xlua` 这个 DLL 中。在运行时，CLR 会在 xlua.dll 编译后的结果中寻找**同名函数**（lua_tothread）并导出它。

`DllImport` 是 .NET Framework 时代的版本，在 .NET7 开始已经变成 `LibraryImport` 这个特性了，因为 xLua 出现比较早，所以还是使用的前者，这里知道它们俩是同一个东西就行了。

### 3. CallingConvention 参数

`CallingConvention.Cdecl` 表示：调用 Native 函数后，由调用方负责恢复栈；还有一个枚举 `CallingConvention.StdCall`，它则表示由被调用者清理栈；这个约定非常重要，如果指定错了，调用栈会没人清理或者被清理两次，导致错误。

### 4. IntPtr 类型

`IntPtr L` 是一个内存指针，对应 Lua 中的 `lua_State* L` 或者指向别的内存地址，在这个函数示例中，返回的是一个 Lua 协程对象的 native 地址。

### 5. DLL 文件在哪儿？

xLua 提供了不同平台的 native 的库，可以将它们放在工程中的任意 Plugins 文件夹下参与编译：

``` title="xLua native dll"
Plugins/
├─ x86_64/
│  └─ xlua.dll
├─ Android/
│  └─ libxlua.so
├─ iOS/
│  └─ libxlua.a
...
```

我们可以看到不同平台的库，后缀不同甚至名称也不同，但它们的内容和实现是相同的。实际上，CLR 在运行 `DllImport("xlua")` 时，会根据平台自动补全文件名和后缀，比如在 Windows 平台会寻找 `xlua.dll`，在 Android 平台（底层 Linux）会寻找 `libxlua.so` 。

### 6. IL2CPP 下还是 P/Invoke 吗？

这是一个很有意思的东西。在 Mono 下，由于实际跑的是 C#，运行时还是 CLR，所以：

``` title="Mono"
C# -> P/Invoke -> libxlua.so
```

还是标准的 P/Invoke 。

而在 IL2CPP 的环境下，实际跑的是 C++ 代码，运行时已经没有 CLR 了。但 `DllImport` 仍然工作，只不过不是经过 CLR 的 P/Invoke Stub 了，而是变成 C++ 层面的外部函数调用：

``` title="IL2CPP"
C# -> IL -> C++ -> libil2cpp.so --直接调用--> libxlua.so
```

本质上已经不是 P/Invoke 了，但在抽象层面我们还是可以当做这里有平台调用。
