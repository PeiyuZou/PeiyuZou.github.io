## 一、大世界一共有哪些重要实现（全貌盘点）

体素大世界要做到"2000 平方公里无缝漫游 + 玩家可实时挖改 + 服务器权威同步 + 移动端跑得动"，拆成这几块：

| # | 模块 | 解决什么 | 关键文件 |
| --- | --- | --- | --- |
| 1 | **TD 分块流式加载** | 世界太大，内存放不下、不可能全加载，只常驻玩家周边一圈并随玩家滚动 | `TiledScheduler.cs`、`MCPatch.cs`、`ISchedulable.cs` |
| 2 | **Chunk 八叉树 + 多 LOD 距离查询** | TD 内部也只显示该距离该看的 LOD；远处粗、近处细 | `Octree.cs`、`MCPatch.UpdateLod`、`ChunkEntity.MergeChunk` |
| 3 | **Greedy Mesh 贪心面合并** | 体素面太多，合并相同相邻面降面数 | `MCChunkRenderData.cs` |
| 4 | **物件 LOD（场景物件，非体素）** | NPC/宠物/家具等按屏高占比切 LOD，远处省开销 | `SceneObjectLodGroup.cs`、`SceneObjectManager.UpdateLod`、`MCScene.UpdateLod` |
| 5 | **服务器增量推送合并** | 服务器权威，只推变化的 chunk，客户端合并进本地八叉树 | `MCPatch.PreProcessDataNet/ProcessNetData`、`MCNetworkManager` |
| 6 | **AOI（物件视野管理）** | 只更新视野内物件的渲染与逻辑 | `AoiManager.cs`（分级距离 AOI） |
| 7 | **客户端插值平滑（时间线）** | 服务器状态同步离散广播，客户端插值重现为平滑动作 | `NetRoleTimeLineComponent`、`NetMoveToBehavior` |
| 8 | **坐标精度（double）+ 相机相对渲染** | 大世界 float 会顶点抖动，用 double + 渲染时平移到相机附近 | `YKVector3d`、`WorldSceneSystem.SetCameraRelativeRender` |
| 9 | **遮挡剔除（体素专用）** | 地下/石头内部物件不渲染，离线烘焙 + 运行时查询，无 GC | `MCOCManager.cs`、`MCOCPatch.cs` |
| 10 | **玩家编辑块 + 向上更新 LOD + 上行** | 玩家挖/放块，改本地数据、通知服务器、刷新各级 LOD | `MCChunkIO.SetBlock/ApplyChangeToParent`、`MCNetworkManager.SetBlock` |
| 11 | **传送 / 跳点加载** | 跨大距离瞬移，四阶段加载完成后回调业务 | `MCNetworkManager.OnJumpToPosition`、`MCScene.SetLocation` |
| 12 | **异步 + 多线程 + 分帧防卡顿** | 加载/解析在子线程，主线程分帧提交 | `MCScene.ThreadRunnable`、`MCManager.Update`（`%3`） |
| 13 | **内存与池化** | Mesh/GameObject 池化复用、不可见 Chunk 跳过解压、AB 引用计数 | `m_ChunkGameObjectPool`、`SceneAssemblerManager.RefCount` |

记忆主线：**外层 TD 滚动进出一个圈（1）→ 每个 TD 内层 Chunk 按距离选 LOD 滚动（2）→ 物件 AOI/LOD 另管视野（4、6）→ 服务器只推变化、客户端合并+插值（5、7）→ 玩家能改并上行（10）→ 全程 double 坐标 + 子线程 + 分帧（8、12）**。

## 二、每块的运作方式（精炼版，快速掌握）

### 1. TD 分块流式加载

- 世界切成大量 **512×512 的 TD**（Terrain Data，`PatchSize=512`）。
- 加载范围 `ScheduleRange` 默认 2048，画质档乘 0.5/1/2。`iSizeX=ceil(2048/512)=4`，所以范围 `[center-4, center+4)`，横竖各 8 个 TD。
- **默认常驻 8×8 = 64 个 TD**（低档 16、超高档 256）。每个 TD 内约 256 个 lod1 chunk。
- `TiledScheduler.PatchScan` 每帧（带节流）按玩家位置算范围：范围外的 TD 进卸载队列分帧释放、范围内但缓存里没有的 TD new 出来进加载队列分帧加载；加载按"离玩家距离"排序，优先中间避免传送穿帮。
- 缓存只在 `ConcurrentDictionary<long,TiledPatch>` 里存"当前常驻的 TD"，超范围即 `TryRemove`。**TD 卸载即销毁数据，重进要重走 IO 重新解析**（见第三部分细讲）。

### 2. Chunk 八叉树 + 多 LOD 距离查询

- 每个 TD 一棵八叉树，根节点 512。
- 5 级 LOD：blockSize = 1/2/4/8/16，每级一套独立 MCWorld。
- `MCPatch.UpdateLod()` 用玩家位置在八叉树上 `OctreeQueryByLod`，得到"这个距离该显示哪些节点"；上一帧在、本帧不在的卸载，本帧新出现的加载（用 `LoadFlag` 区分新旧）。
- **高 LOD 数据由低 LOD 子节点合并而来**：`MergeChunk` 把 8 个子 block 取众数类型当父 block 的类型；带"可见面权重"优化（只统计暴露面多的类型，远处看起来不会被地下块污染）。
- LOD 距离阈值按画质档配置（如低档 lod16 在 1024m、超高档 4096m）。

### 3. Greedy Mesh 贪心面合并

- 每个 Chunk 走贪心面合并：相邻相同材质/类型的面片，按水平、垂直两个方向合并成更大的包围盒面，再按材质类型分桶输出多个 Mesh。
- Lod0 才显示异形块（走 GPU Instancing），高 LOD 只画 Cube 面 → 远处面数极低。

### 4. 物件 LOD（场景物件）

- **判据不是纯距离，是"物件包围盒屏高占比"**（Unity 式 LODGroup 思路）：`percent = BoundSize² / (baseVPHeight² × dist²)`，与每个 entity 配置的 `LODScreenRelativeTransitionHeights[]` 比较得 LOD 级数。
- `baseVPHeight` 由 `GetVisualPlaneDistance(FOV, Aspect)` 算出相机视平面高度，随 FOV 变化自动适应。
- 切换走 `IEntityLod.OnLodChanged(lod)`——根据 LOD 切的是 MeshDraw/skin，不是换 prefab。
- 节流：`LodUpdateDirtyChecker` 每 0.2s 且相机位/FOV/朝向变化才重算。
- 装配分帧：`SceneObjectLodGroup.Update` 每帧从队列按 `MaxLoadingCountPreFrame` 上限出队 `OnLoad`，扫描入/出树在子线程做。

### 5. 服务器增量推送合并

- **服务器只推变化的 chunk 坐标（增量），不是全量快照**。
- 客户端 `PreProcessDataNet` 把变化坐标落到 128m（MaxBlockSize 4 × ChunkSize 32）的桶，登记进 `ServerChangedChunks`（value=false 表示未处理）。
- `ProcessNetData` 遍历 value=false 的，`AddChunkEntityRecursive` 自顶向下八分递归创建/插入 ChunkEntity 直到 blockSize=1，处理完置 true。
- **超范围重入问题**：服务器只维护玩家周边 9×9、半径 `SERVER_LOD0_RANGE=128m` 的 lod0 chunk 并推送变化。当画质档让本地 lod0 加载距离超过 128m 时，玩家走动中离开 128m 又走回来的 chunk，服务器在其出范围期间没推过变化，本地数据已过期 → `IsNeedRefreshLod0ChunkOutServerRange` 检测到该 chunk 中心重新进入 128m AABB，就触发重新向服务器请求这块的最新数据。

### 6. AOI（物件视野管理）

- 实际是**分级距离 AOI**，不是简历里写的"四叉树"。`EAoiLevelType.Center(32²)/Mid(64²)/Out` 三级。
- 服务器推 `home_aoi_change`（增删），客户端先入 `m_JoinCache`，每帧用主角位置判 AOI 级别后才真正入场。越级直接不渲染不表现。
- 八叉树（`Octree`）只服务于体素地形的 LOD，跟 AOI 是两套，别混。

### 7. 客户端插值平滑（时间线）

- **状态同步，非帧同步**。服务器下发离散广播（移动路径、方向、攻击、buff），客户端插值重现。
- `NetRoleTimeLineComponent` 按 `HostTime` 排队播放各类 Action（Idle/Move/MovePath/Turn/Attack/SkillMove/Throw）；移动用 `Lerp(StartPos, DestPosition, lerpTime)` 插值，多点路径按各自 `UseTime` 拼成总时长。
- `NetMoveToBehavior` 用经验时长 0.2s 做移动平滑吸收抖动。
- **没有客户端预测回滚**——是服务器权威 + 时间线插值平滑 + 位置补偿（火车位置补偿）。面试别说成"预测回滚"。

### 8. 坐标精度（double）+ 相机相对渲染

- 逻辑/世界坐标全程用 `YKVector3d`（double），避免远离原点处 float 精度丢失导致顶点抖动。
- 渲染时开启 URP 内置 `useCameraRelativeRender`，把世界美术坐标在渲染矩阵里平移到相机附近，使 GPU 顶点坐标变成相机原点附近的小浮点数。
- 给主角 view 挂 `AnimationOriginShift` 组件做动画原点反向平移，弥补相机相对偏移对动画根位移的影响。

### 9. 遮挡剔除（体素专用）

- **离线烘焙 + 运行时查询**。烘焙粒度 OCCell `(8,4,8)`，每个 OC 文件覆盖 32m，记录"在该 cell 视点下不可见的实体 uid 列表"，GZip 压缩成 `.bytes`。
- 运行时 `MCOCPatch.UpdateOCCells` 每帧按玩家位置与 cell 中心比 AABB 距离，范围内 `Enable()`（解压出 HashSet）、范围外 `Disable()`（归还池）。
- **全程无 GC**：解压用 `ArrayPool<byte>.Shared`，HashSet 用 `HashSetBufferPool<long>` 池化，逐 long 手工拼接避免中间数组。

### 10. 玩家编辑块 + 向上更新 LOD + 上行

- 玩家改块走 `MCNetworkManager.SetBlock`：改本地八叉树 chunk 数据并标脏；同 chunk 改动攒进 `m_SetBlockData`（上行增量缓存，`ConcurrentDictionary`）。
- chunk 在树 → `MarkBlockDirty` 触发回包重建；chunk 不在树 → `PreProcessDataNet` 入 `ServerChangedChunks` 并重新拉取该 chunk 后合并本地待改。
- 向上更新 LOD：`OnChunkModifyUpdateLod` → `MCModifyPatch.UpdateChunkLodUp` 自底向上递归 `MergeChunk` 直到 Size>512 停，保证 lod1/lod2/lod3 数据一致。
- `NeedSave` 是离线/本地编辑存盘标记，联机热路径不走落盘。

### 11. 传送 / 跳点加载

- `MCNetworkManager.OnJumpToPosition` → `MCScene.SetLocation`：重置 LoadingState 后以新坐标为扫描中心重新流式加载。距离>32 才走完整 loading，降低后台加载优先级 / 提速异步上传。
- 四阶段：`TilePatchFinish → ChunkFinish → EntityFinish → All`，`IsLoadingStateFinish()` 判 `state==All` 才回调业务。各阶段还回调 Lua 埋点。

### 12. 异步 + 多线程 + 分帧

- 场景有独立工作线程（`MCScene.ThreadRunnable`），扫描/解压/合并在子线程，渲染提交回主线程。
- 主线程 `MCManager.Update` 按 `m_FrameCount % 3` 分三帧：SetBlock 刷新 / WaitRemove / 遍历脏 Chunk 重建 Mesh。
- LOD 检查节流 0.2s，相机位/朝向/FOV 没动就跳过。资源加载用 `AsyncCombiner` 异步聚合，回调对象池化零 GC。

## 三、面试高频点细讲

### 高频点 A：流式加载到底"流式"在哪 / 重进要不要重 IO / 做没做缓存

（这块我专门追问过，面试官最爱挖）

**"流式"三层含义**：
1. 不预加载全图，只常驻玩家周边一圈 TD；
2. "边走边加载"——`PatchScan` 每帧按当前位置动态算加载列表（不是预定义），玩家北走北边 TD 逐个入、南边逐个出，加载按离玩家距离排序避免传送穿帮；
3. "用完即弃"——卸载是真销毁数据，不是挂着不渲染。
另外是**两级流式**：外层 TD 滚动进出（`TiledScheduler`），内层每个 TD 的 Chunk 又按 LOD 距离在八叉树查询滚动，不可见 Chunk 连解压都跳过。

**重进会重走一次完整 IO**：`OnRelease()` 里 `RemoveAllChunk()` 对每个 ChunkEntity 调 `UnLoad()`（`DataCompress=null; chunk.Dispose(); chunk=null`），再 `m_UserData=null`，**卸载即销毁**；重进 `OnLoad` 走 `SceneAssemblerManager.LoadAsset` 重新加载 `.bytes` 并 `ProcessData` 重新 `BinaryReader` 解析，**重进即重读+重解析**。

**缓存要分层讲**（别被绕进去）：

| 层级 | 缓存？ | 说明 |
| --- | --- | --- |
| TD 字节（AB） | 有引用计数字典，但 TD 卸载时 `UnLoadAsset(path, true)` 主动 Remove 了 | 通常不在缓存里 |
| 解析后 ChunkEntity/八叉树 | **没有**，彻底销毁 | 重进重建 |
| Chunk 的 Mesh/GameObject | **有**，`m_ChunkGameObjectPool` 池化复用 | 上限约 100 |
| 不可见 Chunk | 跳过解压而非清掉 | `EnableInvisibleOptimize` |

**取舍**：没做"TD 卸载后留 LRU 缓存"是刻意的——收益是内存峰值可控（移动端 MMO 硬约束），代价是反复进出同一片区域要反复解析。真要做我会先埋点统计 TD 重进命中率，超阈值再上 LRU，避免过早优化；且脏块/服务器增量的缓存一致性要单独维护。

### 高频点 B：为什么自研而不是用 Unity StreamingWorld

体素世界特点是**会被玩家实时修改且要存盘 + 服务器权威增量同步**——自带 StreamingWorld 处理不了高频率局部重建、Mercurial 面合并、增量推送合并、向上更新 LOD 这几件事，所以自研基于八叉树 + Greedy Mesh + 多 LOD 的一套。

### 高频点 C：怎么支撑 2000 平方公里不掉帧

几个关键凑一起：① TD 分块只常驻玩家周边 64 个；② 5 级 LOD 远处粗降面；③ double 坐标 + 相机相对渲染防抖；④ 子线程扫描解析 + 主线程 `%3` 分帧提交；⑤ 不可见 Chunk 跳过解压；⑥ Mesh/GameObject 池化；⑦ LOD/相机节流避免每帧全算；⑧ AsyncCombiner 异步聚合 + 回调池化零 GC。

### 高频点 D：服务器增量推送合并的关键设计

服务器只推变化的 chunk 坐标（增量）。客户端不立即合并数据，而是登记进 `ServerChangedChunks` 待处理，`ProcessNetData` 用 `AddChunkEntityRecursive` 递归插入八叉树、标记 dirty 重建。关键是**服务器推送范围（128m）与本地 LOD 加载范围解耦**：本地画质高、lod0 看得比 128m 远，那些超出 128m 又走回来的 chunk 服务器没推过变化、本地数据已过期，所以要 `IsNeedRefreshLod0ChunkOutServerRange` 重新请求。这是体素大世界 + 服务器权威特有的"范围不匹配"问题，面试时是个亮点。

### 高频点 E：物件 LOD 为什么用屏高占比而不是纯距离

纯距离没法适应不同 FOV（同一距离，FOV 大物件在屏上更小）。`percent = BoundSize² / (baseVPHeight² × dist²)` 把物件包围盒屏高占比算出来和阈值比，FOV 变化自动适应。节流 0.2s + 相机变化才重算，避免频繁切换抖动。

### 高频点 F：遮挡剔除怎么做无 GC

体素世界地下/石头内部物件多，离线烘焙"每个 OCCell 视点不可见的 uid 列表"成 GZip bytes；运行时按玩家位置激活/休眠 cell，查到不可见集就跳过渲染。无 GC 是因为解压用 `ArrayPool<byte>`、HashSet 用 `HashSetBufferPool<long>` 池化逐 long 拼接，全程不 new 中间数组。

## 附：面试可能被问但代码里要谨慎表述的点

- **AOI**：简历写"四叉树 AOI"不属实，真实是分级距离 AOI（Center/Mid/Out），八叉树只用于地形。表述改"分级距离 AOI"或"AOI + 八叉树（地形用）分两套"。
- **网络同步**：别提"四叉树 AOI""客户端预测回滚"——是状态同步 + 时间线插值平滑 + 位置补偿，无预测回滚。
- **体素编辑器**：代码里是 EntityGen 实体生成器的体素配置工具（占位体素/碰撞体素可视化编辑），不是地图建造编辑器。地图体素编辑走 `MCTerrainModifyTool`。
- **远程自动构建 AB**：无自研 HTTP 远程服务证据，是"打包机命令行远程驱动 Unity 打 AB"。

> 这些坑点面试时心里有数，被追问到具体细节就说"分级距离 AOI"、"时间线插值"、"服务器权威状态同步"这类真实实现的词，别沿用简历里被美化过的措辞。
